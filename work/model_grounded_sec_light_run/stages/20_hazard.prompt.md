# HARA 核心知识

## HARA 边界

HARA 是 ISO 26262 概念阶段的危害分析与风险评估活动，用于识别由 item 功能异常引起的车辆级危险事件，评价 S/E/C 并得到 ASIL 或 QM，随后形成车辆级安全目标。

HARA 不做：

- 组件 FMEA。
- 根因诊断。
- 软件详细设计。
- 安全机制设计。

## Item Definition

Item 定义应说明被分析系统的功能边界、车辆级预期行为、操作模式、外部接口、驾驶员/用户角色、假设和排除项。需求文档没有写清楚的信息必须作为假设保留。

## Malfunction

Malfunction 是预期功能的行为偏离。描述要从车辆/用户可观察行为出发，不要写根因。

常用引导词：

- 功能丧失：应提供功能时完全不提供或无法保持。
- 过大：输出、强度、持续时间、作用量超过预期。
- 过早：功能在合理时机之前发生。
- 过小：输出、强度、持续时间、作用量不足。
- 过晚：功能响应延迟或结束延迟。
- 非预期激活：没有请求或不满足条件时激活，或应保持关闭时激活。
- 卡滞：状态、输出、执行器或显示保持在某一状态无法改变。
- 方向错误：车辆运动、执行方向、显示指示与预期相反。

引导词不是填空游戏，也不是安全风险筛选器。sheet1 阶段先建立功能行为模型，再判断每个引导词对应的偏离是否逻辑存在。只要故障存在，就应进入 sheet1；是否产生车辆级危害和 ASIL 等级在后续阶段判断。

功能行为模型至少包含触发源、允许条件、执行动作、目标状态、物理作用量、时间关系、相反方向/状态和反馈提示。基于这些维度判断功能丧失、过大、过小、过早、过晚、非预期激活、卡滞和方向错误是否存在。

## Vehicle-Level Hazard

整车级危害是车辆层面的危险状态，不是零部件故障。优先使用枚举：

- 纵向运动：非预期加速、失去加速、非预期减速、失去减速、制动能力降低、非预期纵向移动、无法纵向移动、相反方向纵向移动、俯仰/滚动。
- 侧向运动：非预期侧倾、非预期横摆、失去横向运动、侧倾/滚动。
- 垂向运动：非预期垂直运动、横摆/滚动。
- 特殊危害：热事件、电击、视野降低、错误信息、失去报警、其他道路使用者误判、需要保护时失去保护、无危害。

## Hazardous Event

危险事件 = 整车级危害 + 操作场景 + 风险对象/人员 + 可能伤害机制。

有效危险事件必须通过反事实检查：如果移除 item 故障，事件不应以同样方式发生。

## Safety Goal

安全目标是车辆级安全意图，避免或控制危险事件的不可接受风险。它应描述需要防止的危险状态，例如：

- 防止车辆在驻车状态下发生非预期移动。
- 防止夜间行驶时前照明能力丧失导致驾驶员视野不足。

安全目标不应写成：

- 增加传感器冗余。
- 优化算法。
- 检测 CAN 报文丢失。
- 驱动电路诊断。


# Sheet2 故障-整车危害映射思维模型

本指南只用于 `map_hazard` 阶段。sheet2 的目标是把 sheet1 中每一条单独故障映射到车辆级危害，保证追溯完整，同时避免把故障误解释成另一个引导词或另一个场景。

## 核心原则

- sheet1 出现的每一条故障都必须进入 sheet2。若没有车辆级安全危害，填写 `无危害`，不要删除。
- 整车级危害必须是车辆级状态或道路使用者可感知的危险状态，不是零件寿命、维修成本、能耗、法规符合性或舒适性问题。
- 映射时只能使用当前故障本身。不要为了构造危害，额外引入另一个故障，例如把 `输出过大` 当成 `非预期激活`，或把 `方向错误` 当成 `非预期关闭`。
- 备注必须说明故障如何导致整车危害；如果只是可能影响耐久、能耗或用户体验，应明确写 `无危害` 的理由。
- sheet2 可以保留弱危害供 HARA 评估，但备注要提示后续 HARA 的限制条件，例如低暴露、驾驶员可提前发现、低可控性不成立、低严重度等。

## 过大/过小的边界

`过大` 和 `过小` 只描述请求动作已经发生时输出作用量过高或过低，不等同于无请求动作。

- 如果作用量过大直接造成车辆运动、减速、加速、转向或伤害风险，可以映射到对应整车危害。
- 如果作用量过大只会增加零件磨损、热负荷、噪声、能耗或使用寿命下降，通常是 `无危害`。
- 不要把 `EPB夹紧力过大` 写成“低速时非预期拉起导致减速”。低速非预期拉起属于 `EPB非预期拉起`，不是夹紧力过大的危害机制。

## 外部照明/近光灯

近光灯不同于远光灯。近光灯异常可能影响视野或让其他道路使用者误判，但通常不应默认成高速强眩目或高 ASIL。

典型映射：

- `近光灯无法点亮`、`近光灯亮度过小`、`近光灯点亮响应过晚`、`近光灯非预期熄灭`、`近光灯卡滞在熄灭状态`、`近光灯请求点亮却熄灭`：可映射为 `驾驶员视野丢失或者降低`，但备注应提示仅在夜间、隧道、低照度且驾驶员未及时采取措施时成立。
- `近光灯无法熄灭`、`近光灯亮度过大`、`近光灯非预期点亮`、`近光灯卡滞在点亮状态`：可映射为 `其他驾驶员/行人误判`，但备注应提示近光灯影响通常弱于远光灯，HARA 需谨慎评估。
- `近光灯熄灭响应过晚`、`近光灯请求熄灭却点亮`：通常为 `无危害`，因为请求熄灭通常发生在不需要近光灯的状态，驾驶员也容易发现并再次操作。

HARA 提醒：

- 夜间刚起步、傍晚行驶、进入隧道前后，驾驶员通常会主动尝试开灯并能发现照明异常；不要默认成突然不可控。
- 几秒级点亮延迟通常不应直接推成高 ASIL，除非场景明确为低照度中突然进入危险区域且驾驶员无足够反应时间。
- 近光灯非预期点亮不能只按白天判断为无危害；夜间或低照度下仍可能影响其他道路使用者判断，但通常弱于远光灯。
- 方向错误不能简单等同于非预期激活或功能丧失。请求点亮却熄灭时，驾驶员可能立即察觉并采取措施；请求熄灭却点亮时，通常不构成车辆级安全危害。

## EPB/驻车制动

典型映射：

- `EPB无法拉起`、`EPB夹紧力不足`、`EPB拉起响应过晚`、`EPB卡滞在释放状态`、`EPB请求拉起却执行释放`：可映射为 `非预期的纵向移动`，机制通常是坡道、蠕行、外力或驱动扭矩导致车辆移动。
- `EPB非预期拉起`：可映射为 `非预期的减速`，但必须说明故障如何越过静态操作域或发生在低速/行驶状态。
- `EPB夹紧力过大`：静态拉起功能中通常为 `无危害`；主要影响可能是卡钳、制动盘、电机或机构寿命，不能写成非预期减速。


# 功能族模式

只读取与当前系统相关的段落。下面是生成 malfunction、hazard、scenario 的参考，不是强制模板。

## 外部照明/前照灯

典型功能：

- 近光灯、远光灯、位置灯、自动灯光、故障提示、夜间开灯提示。

典型 malfunction：

- 近光灯无法点亮。
- 近光灯无法熄灭。
- 近光灯亮度过大，可能使其他道路使用者误判或眩目；外部照明的车辆级输出天然包含亮度/照度维度，即使需求只描述驱动开/关也要审查。
- 近光灯亮度过小，导致驾驶员视野降低。
- 应关闭时仍点亮或非预期点亮。
- 非预期关闭，尤其是夜间或隧道内由点亮状态变为熄灭状态。
- 近光灯点亮响应过晚。
- 近光灯熄灭响应过晚。
- 近光灯卡滞在关闭状态或卡滞在打开状态。
- 近光灯请求点亮却熄灭，或请求熄灭却点亮。
- 指示灯或提示错误，导致驾驶员误判外部照明状态。

典型整车级危害：

- 驾驶员视野丢失或者降低。
- 其他驾驶员/行人误判。
- 驾驶员获取错误的信息。
- 失去报警。

有效场景机制：

- 夜间、隧道、无路灯或弱照明道路，近光灯无法点亮，驾驶员前方视野不足，未及时发现行人、障碍物或前车。
- 夜间/隧道中近光灯非预期熄灭，驾驶员短时间内视野突然下降。
- 对向来车或前车因异常灯光状态误判本车位置或意图。
- 夜间开灯提示丢失只在驾驶员实际可能忘记开灯且外部照明不足时才有安全相关性。
- 近光灯非预期点亮在白天通常无危害，但在夜间、低照度、会车或复杂交通中可能影响其他道路使用者判断；不要只按白天工况判为无危害。

无效或弱场景：

- 白天正常天气仅近光灯不亮，通常安全影响较弱。
- 把“灯不亮导致违反法规”当作 HARA 人身伤害事件。
- 风险对象不在本车可照明/行驶路径内。
- 把近光灯非预期点亮默认等同于远光灯强眩目；若没有亮度过大或配光错误故障，不要自动推导成高速致命眩目。
- 近光灯无法点亮、亮度过小或卡滞熄灭在夜间刚起步/傍晚行驶中通常容易被驾驶员发现并采取措施，HARA 中 E/C 往往较低，常见场景可能为 QM。
- 近光灯点亮响应过晚若只是短暂延迟，通常不支持高 ASIL，除非场景中驾驶员在低照度下突然失去必要视野且无足够反应时间。
- 近光灯请求点亮却熄灭不能简单等同于非预期熄灭；它是有请求时的方向错误，驾驶员通常更容易发现。
- 近光灯请求熄灭却点亮通常不构成整车级危害；请求熄灭说明当前通常不需要近光灯，且驾驶员容易发现并再次操作。

## 电子驻车制动 EPB/驻车保持

典型功能：

- 静态开关拉起、释放、自动驻车、坡道驻车、休眠唤醒后拉起。

典型 malfunction：

- EPB无法拉起。
- EPB拉起夹紧力过大。
- EPB拉起夹紧力不足。
- EPB拉起响应过晚。
- EPB非预期拉起；未请求、行驶中、不满足允许条件等是同一本质故障的场景/操作域变体，sheet1 不重复拆分。
- EPB卡滞在释放状态导致无法拉起。
- EPB卡滞在拉起状态导致无法释放。
- EPB请求拉起却执行释放，或请求释放却执行拉起。
- 驻车状态反馈错误，驾驶员误判车辆已驻稳。

典型整车级危害：

- 非预期的纵向移动。
- 非预期的减速。
- 无法纵向移动。
- 驾驶员获取错误的信息。

有效场景机制：

- 坡道驻车或停车未熄火时，EPB 请求拉起失败，车辆因坡度、驱动蠕行或外力发生非预期移动，撞击行人、后车、障碍物。
- 静止/低速时 EPB 非预期拉起，引起车辆突然减速或后车追尾。
- 行驶中 EPB 非预期拉起属于低可控性高风险场景；若当前功能只允许静态拉起，必须说明非预期激活如何越过操作域。
- EPB 夹紧状态反馈错误可能导致驾驶员离车，车辆随后移动，驾驶员不在车上时 C 往往更差。
- 驻车拉起失败、夹紧力不足、响应过晚、请求拉起却释放等“非预期纵向移动”场景，必须说明车辆从静止/低速如何获得纵向力：坡度、驱动蠕行、外力、拖拽或释放已有保持力。没有纵向力来源时不能构成整车危害。
- 坡道方向和溜车方向必须一致：默认车辆朝行驶方向停放时，下坡道路失去驻车保持应向前滑移，上坡道路失去驻车保持应向后滑移。若写反方向，必须说明车头朝向、倒车方向、外力或驱动扭矩等原因。
- 驾驶员仍在车上且车辆从静止开始缓慢溜移时，通常可以通过脚刹、P挡、重新操作或观察参照物补救，C 不宜直接给 C3；暴露率也应针对“坡道/行人/外力等组合场景”评估，不把每次停车都等同于 E4。
- 驾驶员已离车、坡度明确、行人或其他车辆处于溜车路径内时，可形成更高 S/C；但 E 仍取决于该组合场景的实际频率。

无效或弱场景：

- EPB夹紧力过大在静态拉起功能中通常只影响制动器、电机、卡钳或机构寿命，不应映射为非预期减速；非预期减速属于 EPB 非预期拉起。
- “雨雪导致车辆溜坡”但没有坡度、驱动扭矩、外力或制动力不足作为纵向力来源。
- 车辆状态为驻车却给出 60 km/h 以上车速。
- 车辆状态写成前向匀速、转弯、变道等行驶状态，但车速为静止状态。
- 静态开关拉起功能的故障被直接扩展为高速巡航故障，除非故障模式明确为非预期激活且解释越域。
- 把 Released 状态请求拉起失败、休眠唤醒失败等不同前置条件重复写成多个功能丧失故障；sheet1 应概括为“驻车时 EPB 无法拉起”，前置条件在 sheet2/HARA 展开。
- 驾驶员主动拉起开关通常没有“过早”故障；请求前拉起属于非预期激活。

## 制动系统

典型危害包括失去减速、制动能力降低、非预期减速。核心场景关注前车距离、速度、路面附着、弯道/直道、后车距离和驾驶员可用反应时间。

## 转向系统

典型危害包括失去横向运动、非预期横摆、方向错误。核心场景关注弯道、变道、车道保持、对向车辆、道路边界和驾驶员施力能力。

## 动力/驱动系统

典型危害包括非预期加速、失去加速、相反方向纵向移动。核心场景关注起步、超车、路口、拥堵跟车、坡道、倒车和驾驶员可通过制动/换挡控制的能力。

## HMI/提示/报警

HMI 本身通常不是直接伤害源。只有当错误信息、误报警、漏报警会改变驾驶员决策并导致车辆进入危险状态时，才形成危险事件。必须说明驾驶员基于错误信息采取了什么动作。
