# HARA 核心知识

## HARA 边界

HARA 是 ISO 26262 概念阶段的危害分析与风险评估活动，用于识别由 item 功能异常引起的车辆级危险事件，评价 S/E/C 并得到 ASIL 或 QM，随后形成车辆级安全目标。

HARA 不做：

- 组件 FMEA。
- 根因诊断。
- 软件详细设计。
- 安全机制设计。

## Item Definition

Item 定义应说明被分析系统的功能边界、车辆级预期行为、操作模式、外部接口、驾驶员/用户角色、假设和排除项。需求文档没有写清楚的信息必须作为假设保留。

## Malfunction

Malfunction 是预期功能的行为偏离。描述要从车辆/用户可观察行为出发，不要写根因。

常用引导词：

- 功能丧失：应提供功能时完全不提供或无法保持。
- 过大：输出、强度、持续时间、作用量超过预期。
- 过早：功能在合理时机之前发生。
- 过小：输出、强度、持续时间、作用量不足。
- 过晚：功能响应延迟或结束延迟。
- 非预期激活：没有请求或不满足条件时激活，或应保持关闭时激活。
- 卡滞：状态、输出、执行器或显示保持在某一状态无法改变。
- 方向错误：车辆运动、执行方向、显示指示与预期相反。

引导词不是填空游戏，也不是安全风险筛选器。sheet1 阶段先建立功能行为模型，再判断每个引导词对应的偏离是否逻辑存在。只要故障存在，就应进入 sheet1；是否产生车辆级危害和 ASIL 等级在后续阶段判断。

功能行为模型至少包含触发源、允许条件、执行动作、目标状态、物理作用量、时间关系、相反方向/状态和反馈提示。基于这些维度判断功能丧失、过大、过小、过早、过晚、非预期激活、卡滞和方向错误是否存在。

## Vehicle-Level Hazard

整车级危害是车辆层面的危险状态，不是零部件故障。优先使用枚举：

- 纵向运动：非预期加速、失去加速、非预期减速、失去减速、制动能力降低、非预期纵向移动、无法纵向移动、相反方向纵向移动、俯仰/滚动。
- 侧向运动：非预期侧倾、非预期横摆、失去横向运动、侧倾/滚动。
- 垂向运动：非预期垂直运动、横摆/滚动。
- 特殊危害：热事件、电击、视野降低、错误信息、失去报警、其他道路使用者误判、需要保护时失去保护、无危害。

## Hazardous Event

危险事件 = 整车级危害 + 操作场景 + 风险对象/人员 + 可能伤害机制。

有效危险事件必须通过反事实检查：如果移除 item 故障，事件不应以同样方式发生。

## Safety Goal

安全目标是车辆级安全意图，避免或控制危险事件的不可接受风险。它应描述需要防止的危险状态，例如：

- 防止车辆在驻车状态下发生非预期移动。
- 防止夜间行驶时前照明能力丧失导致驾驶员视野不足。

安全目标不应写成：

- 增加传感器冗余。
- 优化算法。
- 检测 CAN 报文丢失。
- 驱动电路诊断。

## JSON 顶层结构

```json
{
  "project": {
    "system_name": "EPB",
    "source_file": "EPB.docx",
    "item_definition": "系统边界和功能摘要",
    "source_functions": ["静态开关拉起"],
    "assumptions": ["缺失信息或保守假设"]
  },
  "derive_mf": [],
  "mf_vehicle_hazards": [],
  "HARA": [],
  "sg_sum": []

```

sheet 数组内对象的键名必须与下列字段完全一致。

## sheet1: derive_mf

字段顺序：

- `No.`
- `子功能`
- `功能丧失`
- `过大`
- `过早`
- `过小`
- `过晚`
- `非预期激活`
- `卡滞`
- `方向错误`

规则：

- `No.` 为 `系统名_fc编号`，例如 `EPB_fc01`。
- `子功能` 必须优先来自源文档功能清单表中的功能名称。若源文档功能清单只有一个功能，例如 `近光灯`，则 sheet1 通常只有一行，不要把“亮灭逻辑、语音提示、故障检测”拆成多行，除非功能清单明确列出它们是独立功能。
- 每个非空引导词值以 `MF` 编号开头，例如 `MF101 近光灯无法按驾驶员请求点亮`。
- 编号规则：`MF + 功能序号 + 当前功能中非空引导词序号`。例如第 1 个功能的第 1 个非空引导词是 `MF101`，第 1 个功能的第 2 个非空引导词是 `MF102`。
- 一个单元格可包含同一 MF 编号的多个故障，用 `；`、`;`、`或` 分隔，但 sheet2 必须拆分成独立条目。
- sheet1 故障描述只写功能级偏离，不写根因、场景、危害后果或安全机制。
- sheet1 不按安全相关性过滤故障。只要故障在当前功能行为模型中逻辑存在，就应写入；后续可映射为 `无危害` 或 QM。
- 故障描述粒度保持在功能级：同一偏离不要按电源档、状态、休眠唤醒、道路、天气或车速重复拆分。
- 每个引导词是否适用，应由当前功能的触发源、允许条件、执行动作、目标状态、作用量、时序和方向关系推导。

## sheet2: mf_vehicle_hazards

字段顺序：

- `No.`
- `Milf_ID`
- `故障描述`
- `整车级危害`
- `备注`

规则：

- `Milf_ID` 为 `系统名_Milf_编号`，例如 `EPB_Milf_001`。
- `故障描述` 必须复用 sheet1 中的 MF 编号和具体故障描述。
- `整车级危害` 只能来自 `assets/vehicle_hazards.json`。

## sheet3: HARA

字段顺序：

- `List_No`
- `MF_ID`
- `故障描述`
- `整车危害`
- `道路类型`
- `道路条件`
- `环境条件`
- `车辆状态`
- `车速(km/h)`
- `特殊要素`
- `附加条件`
- `驾驶员是否在车上`
- `危害事件`
- `E-解释`
- `暴露频率'E'`
- `有风险的人员`
- `可能的后果('S'的理由)`
- `Severity 'S'`
- `C-解释`
- `控制能力 'C'`
- `结果ASIL`
- `安全目标`
- `安全状态`
- `FTTI(ms)`

规则：

- `MF_ID` 为 `系统名_MF_编号`，例如 `EPB_MF_01`。
- `故障描述`、`整车危害` 必须和 sheet2 对应。
- 场景字段必须来自 `assets/operation_scenarios.json`。
- 每个非 `无危害` 的 sheet2 故障通常应有不少于 3 条 HARA 场景，覆盖最高严重度、高暴露、低可控性/边界工况等不同风险机制；复杂故障应有 4-6 条。不能只给一两条场景就进入 SEC。
- `暴露频率'E'` 取 `E0/E1/E2/E3/E4`。
- `Severity 'S'` 取 `S0/S1/S2/S3`。
- `控制能力 'C'` 取 `C0/C1/C2/C3`。
- `结果ASIL` 必须由脚本根据 S/E/C 计算。
- ASIL 为 `QM` 时，`安全目标`、`安全状态`、`FTTI(ms)` 为空。

## sheet4: sg_sum

字段顺序：

- `SG_No`
- `安全目标`
- `ASIL Level`
- `安全状态`
- `操作模式`
- `FTTI(ms)`

规则：

- `SG_No` 为 `SG001` 形式。
- 每个故障在 HARA sheet 中最高 ASIL 不为 `QM` 时，形成或合并到一条 `sg_sum`。
- `ASIL Level` 取 `A/B/C/D`，继承覆盖条目的最高 ASIL。
- `安全目标`、`安全状态`、`FTTI(ms)` 应来自 HARA 中最高 ASIL 条目。
- `操作模式` 取 `驻车`、`行车`、`驻车和行车`。
