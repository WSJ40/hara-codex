# HARA 工作流与上下文管理

## 目标

本 skill 用于从功能需求文档生成可评审的 HARA Excel。质量优先级：

1. 不漏掉需求文档中的功能和安全相关故障。
2. 不接受故障不是必要原因的危险事件。
3. 不接受物理、车辆状态、速度、风险对象位置互相矛盾的场景。
4. 不让模型手算或猜测 ASIL。
5. 非 QM 风险必须可追溯到安全目标。

## 上下文隔离

不要把“分阶段”理解成同一 Claude 会话里逐步读文件。那样总上下文仍会累积，SEC 阶段仍可能带着 malfunction 推理、候选场景草稿和被拒绝场景。必须优先使用 `scripts/claude_hara_pipeline.py`，每个阶段启动独立 Claude 调用，并用 JSON 文件作为唯一记忆。

最小隔离边界：

1. `item_functions`：只读源文档抽取、功能清单、HARA 核心定义。
2. `derive_mf:<function>`：只读 item 摘要、当前功能片段、引导词定义、相关功能族模式。
3. `map_hazard:<function>`：只读当前功能的 malfunction 和整车危害枚举。
4. `scenario_generate:<fault>`：只读一个故障、一个整车危害、当前功能片段、场景枚举、场景规则；禁止读取 SEC 指南。
5. `scenario_review:<fault>`：只读候选场景和 review rules；不能读取生成器的自由推理。
6. `sec_rate:<scenario>`：只读一个已接受场景和 SEC 指南；不能读取被拒绝候选场景或早期引导词推理。
7. `safety_goal:<fault-group>`：只读非 QM rated scenarios、HARA 核心定义、安全目标规则。
8. `final_review`：只读最终 JSON 和 review rules，运行工具校验。

多功能需求文档必须分片处理：

1. 读取功能清单和目录。
2. 对每个功能，只读取该功能小节、共用 item 边界、必要的接口/状态描述。
3. 生成该功能的 malfunction 和 hazard 后立刻写入 JSON 草稿。
4. 进入下一个功能前，总结已确认的 item 级事实，不携带上一功能的细节。
5. 场景生成按“一个故障/危害一轮”处理，最多同时看同一故障的候选场景。
6. SEC 评级时只读单条已接受场景和 `sec-guide.md`，不要重新读取整份需求文档、malfunction 草稿或场景生成上下文。

## 阶段质量门

### Item 定义

- 系统边界必须来自文档或明确假设。
- 操作模式、电源档位、车辆状态必须来自文档或保守假设。
- 不清楚时写入假设，不要隐式补全。

### Malfunction

- 故障必须是功能行为偏离，不是零件失效根因。
- `derive_mf.子功能` 必须优先使用源文档功能清单中的功能名；不要把功能内部逻辑、告警、状态上报自动拆成独立功能，除非源文档功能清单明确列出。
- 每个故障必须能回答：正常功能是什么、偏离是什么、车辆表现是什么。
- 无安全影响的故障可以保留，但后续应映射为 `无危害`。

### 整车级危害

- 危害必须是车辆级状态，例如“非预期的减速”“驾驶员视野丢失或者降低”。
- 组件级、信号级、DTC、报文错误不是整车级危害。

### 场景

- 场景必须在功能可发生的操作域内；非预期激活可以发生在预期操作域外，但要说明为什么。
- 每个安全相关故障先生成 4-8 条候选场景，再经评审去重和过滤；最终 HARA 中每个非 `无危害` 故障通常不少于 3 条场景。
- 少于 3 条时，必须在 `附加条件` 或项目假设中说明为什么该故障操作域非常窄、其他候选场景被合理排除。
- 风险对象必须位于危险路径内。
- 天气、低附着、夜间、拥堵只能作为放大因素，不能替代故障因果。
- 类似场景只保留 ASIL 最高或机制最有代表性的一条。

### S/E/C

- S 只看伤害严重性，不看发生概率或可控性。
- E 只看该操作场景暴露频率，不看伤害严重性。
- C 只看典型人员是否能避免伤害，不用 ASIL 反推。
- 理由必须引用场景事实。

### Safety Goal

- 安全目标是车辆级安全意图，不是技术方案。
- 可合并相同安全意图的危险事件，但继承最高 ASIL。
- 每个非 QM 最高风险故障都要在 `sg_sum` 中覆盖。

## 迭代策略

如果脚本校验失败，先修复确定性问题：字段、枚举、ID、ASIL、追溯。再修复工程质量问题：场景因果、SEC 理由、安全目标。最后重新导出。


# 功能族模式

只读取与当前系统相关的段落。下面是生成 malfunction、hazard、scenario 的参考，不是强制模板。

## 外部照明/前照灯

典型功能：

- 近光灯、远光灯、位置灯、自动灯光、故障提示、夜间开灯提示。

典型 malfunction：

- 应点亮时不亮或亮度不足。
- 应关闭时仍点亮或非预期点亮。
- 点亮/熄灭延迟。
- 灯光状态卡滞。
- 指示灯或提示错误，导致驾驶员误判外部照明状态。

典型整车级危害：

- 驾驶员视野丢失或者降低。
- 其他驾驶员/行人误判。
- 驾驶员获取错误的信息。
- 失去报警。

有效场景机制：

- 夜间、隧道、无路灯或弱照明道路，近光灯无法点亮，驾驶员前方视野不足，未及时发现行人、障碍物或前车。
- 夜间/隧道中近光灯非预期熄灭，驾驶员短时间内视野突然下降。
- 对向来车或前车因异常灯光状态误判本车位置或意图。
- 夜间开灯提示丢失只在驾驶员实际可能忘记开灯且外部照明不足时才有安全相关性。

无效或弱场景：

- 白天正常天气仅近光灯不亮，通常安全影响较弱。
- 把“灯不亮导致违反法规”当作 HARA 人身伤害事件。
- 风险对象不在本车可照明/行驶路径内。

## 电子驻车制动 EPB/驻车保持

典型功能：

- 静态开关拉起、释放、自动驻车、坡道驻车、休眠唤醒后拉起。

典型 malfunction：

- 请求拉起时未拉起或夹紧力不足。
- 拉起过晚。
- 未请求时拉起。
- 行驶中非预期拉起。
- 状态卡滞导致无法释放或无法拉起。
- 驻车状态反馈错误，驾驶员误判车辆已驻稳。

典型整车级危害：

- 非预期的纵向移动。
- 非预期的减速。
- 无法纵向移动。
- 驾驶员获取错误的信息。

有效场景机制：

- 坡道驻车或停车未熄火时，EPB 请求拉起失败，车辆因坡度、驱动蠕行或外力发生非预期移动，撞击行人、后车、障碍物。
- 静止/低速时 EPB 非预期拉起，引起车辆突然减速或后车追尾。
- 行驶中 EPB 非预期拉起属于低可控性高风险场景；若当前功能只允许静态拉起，必须说明非预期激活如何越过操作域。
- EPB 夹紧状态反馈错误可能导致驾驶员离车，车辆随后移动，驾驶员不在车上时 C 往往更差。

无效或弱场景：

- “雨雪导致车辆溜坡”但没有坡度、驱动扭矩、外力或制动力不足作为纵向力来源。
- 车辆状态为驻车却给出 60 km/h 以上车速。
- 静态开关拉起功能的故障被直接扩展为高速巡航故障，除非故障模式明确为非预期激活且解释越域。

## 制动系统

典型危害包括失去减速、制动能力降低、非预期减速。核心场景关注前车距离、速度、路面附着、弯道/直道、后车距离和驾驶员可用反应时间。

## 转向系统

典型危害包括失去横向运动、非预期横摆、方向错误。核心场景关注弯道、变道、车道保持、对向车辆、道路边界和驾驶员施力能力。

## 动力/驱动系统

典型危害包括非预期加速、失去加速、相反方向纵向移动。核心场景关注起步、超车、路口、拥堵跟车、坡道、倒车和驾驶员可通过制动/换挡控制的能力。

## HMI/提示/报警

HMI 本身通常不是直接伤害源。只有当错误信息、误报警、漏报警会改变驾驶员决策并导致车辆进入危险状态时，才形成危险事件。必须说明驾驶员基于错误信息采取了什么动作。


# HARA 评审规则

## Blocking 规则

以下问题必须修复后才能交付：

- 字段缺失、sheet 名错误、枚举不在允许值中。
- `derive_mf.子功能` 偏离源文档功能清单，或把功能内部逻辑误拆成独立功能。
- MF、Milf、HARA、SG 编号重复或不符合规则。
- sheet2 故障描述不能追溯到 sheet1。
- HARA 故障/整车危害不能追溯到 sheet2。
- ASIL 与 S/E/C 矩阵不一致。
- 非 QM HARA 行缺少安全目标、安全状态或 FTTI。
- QM HARA 行填写了安全目标、安全状态或 FTTI。
- 危险事件中故障不是必要原因。
- 车辆状态与车速明显矛盾，例如驻车 + 高速。
- 场景超出功能操作域且没有合理越域故障解释。
- 风险对象不在车辆危险路径内。
- 非 QM 条目没有进入 `sg_sum`。
- 非 `无危害` 故障的有效 HARA 场景少于 3 条且没有明确排除理由。

## Warning 规则

以下问题应尽量修复，若保留需要在评审报告说明：

- 使用 `ALL` 或 `不涉及` 过多，导致场景不可评审。
- 只通过改变天气、道路名或车速生成重复场景。
- 同一故障场景没有覆盖常见高暴露、最高合理严重度、低可控性/边界工况。
- `E-解释` 没有说明频率或使用占比。
- `可能的后果('S'的理由)` 没有说明伤害对象和伤害机制。
- `C-解释` 没有说明感知、反应时间、空间或驾驶员是否在车上。
- 安全目标包含实现词：传感器、算法、报文、CAN、驱动、诊断、DTC、软件、硬件、冗余。
- FTTI 只有数字但没有基于场景或危害的解释来源。

## 反模式

### AP-001 天气替代因果

雨、雪、雾、夜间、低附着不能单独造成 item 故障导致的危害。它们只能作为暴露或严重度/可控性的影响因素。

### AP-002 驻车高车速

车辆状态为驻车、停车熄火、停车未熄火时，车速通常应为静止状态或低速蠕行。若写高速，必须是非预期移动或功能越域并有解释。

### AP-003 风险对象错位

危险事件中的受伤人员必须位于危险路径内。前方障碍物不能支持“对向车辆乘员受伤”，后方来车不能支持“行人被前照灯照明不足撞击”。

### AP-004 普通交通风险冒充功能安全风险

如果没有 item 故障，同样场景仍会发生同样危险事件，则不是有效 HARA 危险事件。

### AP-005 安全目标设计化

安全目标不能写成具体实现机制。设计机制应出现在功能安全概念或技术安全需求中，不应作为 SG。

## 最终评审提问

逐行检查：

1. 故障描述是否来自正常功能偏离？
2. 整车危害是否是车辆级状态？
3. 场景是否在功能可发生的范围内？
4. 移除故障后，危险事件是否消失？
5. 场景字段之间是否互相一致？
6. S/E/C 理由是否只使用当前场景事实？
7. ASIL 是否由工具计算？
8. SG 是否覆盖最高 ASIL 事件？
