# HARA 核心知识

## HARA 边界

HARA 是 ISO 26262 概念阶段的危害分析与风险评估活动，用于识别由 item 功能异常引起的车辆级危险事件，评价 S/E/C 并得到 ASIL 或 QM，随后形成车辆级安全目标。

HARA 不做：

- 组件 FMEA。
- 根因诊断。
- 软件详细设计。
- 安全机制设计。

## Item Definition

Item 定义应说明被分析系统的功能边界、车辆级预期行为、操作模式、外部接口、驾驶员/用户角色、假设和排除项。需求文档没有写清楚的信息必须作为假设保留。

## Malfunction

Malfunction 是预期功能的行为偏离。描述要从车辆/用户可观察行为出发，不要写根因。

常用引导词：

- 功能丧失：应提供功能时完全不提供或无法保持。
- 过大：输出、强度、持续时间、作用量超过预期。
- 过早：功能在合理时机之前发生。
- 过小：输出、强度、持续时间、作用量不足。
- 过晚：功能响应延迟或结束延迟。
- 非预期激活：没有请求或不满足条件时激活，或应保持关闭时激活。
- 卡滞：状态、输出、执行器或显示保持在某一状态无法改变。
- 方向错误：车辆运动、执行方向、显示指示与预期相反。

## Vehicle-Level Hazard

整车级危害是车辆层面的危险状态，不是零部件故障。优先使用枚举：

- 纵向运动：非预期加速、失去加速、非预期减速、失去减速、制动能力降低、非预期纵向移动、无法纵向移动、相反方向纵向移动、俯仰/滚动。
- 侧向运动：非预期侧倾、非预期横摆、失去横向运动、侧倾/滚动。
- 垂向运动：非预期垂直运动、横摆/滚动。
- 特殊危害：热事件、电击、视野降低、错误信息、失去报警、其他道路使用者误判、需要保护时失去保护、无危害。

## Hazardous Event

危险事件 = 整车级危害 + 操作场景 + 风险对象/人员 + 可能伤害机制。

有效危险事件必须通过反事实检查：如果移除 item 故障，事件不应以同样方式发生。

## Safety Goal

安全目标是车辆级安全意图，避免或控制危险事件的不可接受风险。它应描述需要防止的危险状态，例如：

- 防止车辆在驻车状态下发生非预期移动。
- 防止夜间行驶时前照明能力丧失导致驾驶员视野不足。

安全目标不应写成：

- 增加传感器冗余。
- 优化算法。
- 检测 CAN 报文丢失。
- 驱动电路诊断。

## 外部照明/前照灯

典型功能：

- 近光灯、远光灯、位置灯、自动灯光、故障提示、夜间开灯提示。

典型 malfunction：

- 应点亮时不亮或亮度不足。
- 应关闭时仍点亮或非预期点亮。
- 点亮/熄灭延迟。
- 灯光状态卡滞。
- 指示灯或提示错误，导致驾驶员误判外部照明状态。

典型整车级危害：

- 驾驶员视野丢失或者降低。
- 其他驾驶员/行人误判。
- 驾驶员获取错误的信息。
- 失去报警。

有效场景机制：

- 夜间、隧道、无路灯或弱照明道路，近光灯无法点亮，驾驶员前方视野不足，未及时发现行人、障碍物或前车。
- 夜间/隧道中近光灯非预期熄灭，驾驶员短时间内视野突然下降。
- 对向来车或前车因异常灯光状态误判本车位置或意图。
- 夜间开灯提示丢失只在驾驶员实际可能忘记开灯且外部照明不足时才有安全相关性。

无效或弱场景：

- 白天正常天气仅近光灯不亮，通常安全影响较弱。
- 把“灯不亮导致违反法规”当作 HARA 人身伤害事件。
- 风险对象不在本车可照明/行驶路径内。

## 电子驻车制动 EPB/驻车保持

典型功能：

- 静态开关拉起、释放、自动驻车、坡道驻车、休眠唤醒后拉起。

典型 malfunction：

- 请求拉起时未拉起或夹紧力不足。
- 拉起过晚。
- 未请求时拉起。
- 行驶中非预期拉起。
- 状态卡滞导致无法释放或无法拉起。
- 驻车状态反馈错误，驾驶员误判车辆已驻稳。

典型整车级危害：

- 非预期的纵向移动。
- 非预期的减速。
- 无法纵向移动。
- 驾驶员获取错误的信息。

有效场景机制：

- 坡道驻车或停车未熄火时，EPB 请求拉起失败，车辆因坡度、驱动蠕行或外力发生非预期移动，撞击行人、后车、障碍物。
- 静止/低速时 EPB 非预期拉起，引起车辆突然减速或后车追尾。
- 行驶中 EPB 非预期拉起属于低可控性高风险场景；若当前功能只允许静态拉起，必须说明非预期激活如何越过操作域。
- EPB 夹紧状态反馈错误可能导致驾驶员离车，车辆随后移动，驾驶员不在车上时 C 往往更差。

无效或弱场景：

- “雨雪导致车辆溜坡”但没有坡度、驱动扭矩、外力或制动力不足作为纵向力来源。
- 车辆状态为驻车却给出 60 km/h 以上车速。
- 静态开关拉起功能的故障被直接扩展为高速巡航故障，除非故障模式明确为非预期激活且解释越域。

## 制动系统

典型危害包括失去减速、制动能力降低、非预期减速。核心场景关注前车距离、速度、路面附着、弯道/直道、后车距离和驾驶员可用反应时间。

## 转向系统

典型危害包括失去横向运动、非预期横摆、方向错误。核心场景关注弯道、变道、车道保持、对向车辆、道路边界和驾驶员施力能力。

## 动力/驱动系统

典型危害包括非预期加速、失去加速、相反方向纵向移动。核心场景关注起步、超车、路口、拥堵跟车、坡道、倒车和驾驶员可通过制动/换挡控制的能力。

## HMI/提示/报警

HMI 本身通常不是直接伤害源。只有当错误信息、误报警、漏报警会改变驾驶员决策并导致车辆进入危险状态时，才形成危险事件。必须说明驾驶员基于错误信息采取了什么动作。

10: 
11: 1.2.1 系统工作原理
12: 
13: 在任意电源档位下，通过切换组合开关上的灯光挡位来亮灭近光灯。前车身控制器作为信号采集和处理的主控，接收组合开关发送的灯光档位信号来控制近光灯的亮灭。若为美版车，则仪表显示系统还需要同步点亮灯光总开关指示灯。
14: 
15: 1.2.2 功能逻辑
16: 
17: 1.2.2.1 近光灯亮灭逻辑
18: 
19: ● 输出控制描述：
20: 
21: （1）控制近光灯熄灭：近光灯状态为熄灭，断开近光灯驱动，发送“近光灯”报文（0xXXX，X.X-X.X,0x2）。
22: 
23: （2）控制近光灯点亮：近光灯状态为点亮，输出近光灯驱动，发送“近光灯”报文（0xXXX，X.X-X.X,0x1）。
24: 
25: ● 逻辑描述：
26: 
27: （1）模块得常电需要实时记忆近光灯状态（点亮、熄灭），断电不记忆。
28: 
29: （2）满足下面任意条件，控制近光灯点亮：
30: 
31: （a）近光灯档：接收到1帧“灯光旋钮当前机械结构处于的档位”报文（0x133，4.3-4.5，0x4），或掉线时判断为近光灯档。
32: 
33: （b）AUTO档：根据自动灯光逻辑，判断为近光灯点亮时。
34: 
35: （3）满足下面任意条件，控制近光灯熄灭：
36: 
37: （a）非近光灯/AUTO档：接收到1帧“灯光旋钮当前机械结构处于的档位”报文（0x133，4.3-4.5，0x1/0x3）。
38: 
39: （b）AUTO档：根据自动灯光逻辑，判断为近光灯熄灭时。
40: 
41: （4）掉线处理：连续5s未接收到“灯光旋钮当前机械结构处于的档位”报文（0x133，4.3-4.5）时，认为掉线，执行如下：
42: 
43: （a）ON档电下掉线，若此时采集到近光灯开启/关闭信号为低电平，认为此时处于近光灯档。
44: 
45: （b）ON档电下掉线，若此时采集到近光灯开启/关闭信号为非低电平，默认保持上一状态。
46: 
47: （c）OFF档电下掉线，默认保持上一状态。
48: 
49: （5）整车处于防盗设定状态/防盗状态时，不处理灯光开关的变化（判断“灯光旋钮当前机械结构处于的档位”报文）。
50: 
51: （6）模块上常电默认灯光旋钮的左灯光档当前处于OFF档，近光灯状态为熄灭，断开近光灯驱动，发送“近光灯”报文（0xXXX，X.X-X.X,0x2）。
52: 
53: （7）模块在不掉常电下出现自复位时，近光灯状态需要保持自复位前的状态，然后再按正常逻辑执行。
54: 
55: 1.2.2.2 灯光总开关指示灯逻辑
56: 
57: 若为美版出口车，在控制近光灯亮灭时，需控制仪表显示系统的灯光总开关指示灯同步亮灭。非美版出口车，无此指示灯。
58: 
59: 仪表自身会要求对美版出口车做区分。
60: 
61: 1.2.2.3 夜间行车语音提示功能
62: 
63: （1）同时满足下面所有条件，按照事件周期格式先以短周期发送三帧“夜间行车开灯语音提示”报文（0xXXX，X.X，0x1），然后再发送“夜间行车开灯语音提示”报文（0xXXX，X.X，0x0）：
64: 
65: （a）电源档位为ON档。
66: 
67: （b）灯光开关处于OFF档或小灯档：接收到1帧组合开关发送的“灯光旋钮当前机械结构处于的档位”报文（0x133，4.3-4.5，0x1/0x3）。
68: 
69: （c）车速≥5km/h：接收到1帧“车速信号”报文（0x121，1.0-2.3，车速≥5）。
70: 
71: （d）外界光强档位为5档：发送“光照强度值”CAN报文（0xXXX，X.X-X.X，0x5）时。
72: 
73: （2）不满足上面任一条件（（a）-（d）），发送“夜间行车开灯语音提示”报文（0xXXX，X.X，0x0）。
74: 
75: （3）掉线处理：连续5s未接收到“灯光旋钮当前机械结构处于的档位”报文时，认为组合开关掉线，报文默认保持上一状态；连续10帧未接收到“车速信号”报文时，认为ESP掉线，报文默认保持上一状态。
76: 
77: （4）模块上常电默认发送“夜间行车开灯语音提示”报文（0xXXX，X.X，0x0）。
78: 
79: 1.2.2.4 近光灯故障检测
80: 
81: 当前域需要输出近光灯驱动时，需要对近光灯做故障检测。
82: 
83: （1）在近光灯工作时，满足下面任一条件，认为近光灯故障，发送“近光灯故障状态”报文（0xXXX，X.X-X.X，0x2）：
84: 
85: （a）诊断当前故障有“近光灯驱动电路对地短路故障”。
86: 
87: （b）诊断当前故障有“近光灯驱动电路断路故障”。
88: 
89: （2）在近光灯工作时，满足下面所有条件，认为近光灯正常，发送“近光灯故障状态”报文（0xXXX，X.X-X.X，0x1）：
90: 
91: （a）诊断当前故障无“近光灯驱动电路对地短路故障”。
92: 
93: （b）诊断当前故障无“近光灯驱动电路断路故障”。
94: 
95: （3）在近光灯不工作时，默认近光灯正常。
96: 
97: （4）模块得常电或者休眠唤醒时，默认近光灯正常。
