# S/E/C 和 ASIL 指南

## 输出方式和判断模型

SEC 评级是对当前危险事件的首次工程判断，不是后处理修订。输出理由中禁止出现“评级校正”“修正”“按规则压低/提高”“根据指南调整”等措辞。

每一项理由都必须从本行场景事实推出，不要照抄规则文本：

- S：说明风险对象、碰撞/伤害机制、速度或能量依据；不要用 E 或 C 反推 S。
- E：说明这类操作场景在合理预期使用中的暴露频率；不要把故障发生概率算进 E。
- C：说明驾驶员或相关人员如何感知异常、可用反应时间、可用操作、道路空间、驾驶员是否在控制回路、其他交通参与者是否能避让。

SEC 理由不得引入本行场景字段和附加条件中没有的时序、操作状态或环境条件。若 E 解释使用“起步”“傍晚开灯”“进入暗环境”等前提，场景的 `车辆状态`、`环境条件`、`特殊要素` 或 `附加条件` 必须已经体现这些前提；否则必须围绕已有场景给出暴露理由，例如“夜间无路灯城市道路行驶且驾驶员因刚接手车辆、误以为已开灯等原因未及时发现近光灯未点亮”。

如果正确评级依赖某个更具体的场景前提，应优先在场景生成/评审阶段把该前提写入场景，而不是在 SEC 解释阶段临时补充。

对于同一故障，不同场景可以有不同评级；不要为了保持同一故障评级一致而忽略场景差异。对于低风险场景，要明确说明为什么是低暴露、低严重度或较易控制，而不是删除低风险场景。

## Severity S

S0：无人员伤害。仅舒适性、可用性、财产损失或信息体验问题。

S1：轻度和中度伤害。通常为低速、低能量事件，可能导致擦伤、挫伤、轻微脑震荡、轻度骨折等，一般可恢复。

S2：严重和危及生命的伤害，但通常仍有生存可能。可能包含严重骨折、内脏损伤、严重头部创伤、多根肋骨骨折、脊柱损伤等。

S3：危及生命和致命伤害。高速严重碰撞、翻滚、火灾、电击、车辆失控撞击弱势交通参与者等可评为 S3。

速度参考只作为锚点，不要机械套用：

- 车车正碰/追尾：小于 4 km/h 常为 S0；4-20 km/h 常为 S1；20-40 km/h 常为 S2；大于等于 40 km/h 可为 S3。
- 车车侧碰：小于 2 km/h 常为 S0；2-8 km/h 常为 S1；8-16 km/h 常为 S2；大于等于 16 km/h 可为 S3。
- 与行人、自行车、电动车、摩托车等弱势交通参与者碰撞：0 为 S0；0-8 km/h 常为 S1；8-15 km/h 常为 S2；大于等于 15 km/h 可为 S3。

## Exposure E

E0：不可置信。场景不属于合理预期使用。

E1：极低概率。通常小于 0.4 小时/年，或绝大多数驾驶员一年不到一次。

E2：低概率。0.4-4 小时/年，或一年几次。例如平行泊车、坡道起步、少量雨雪、学校区域。

E3：中等概率。4-40 小时/年，或月度出现。例如倒车、停车场、住宅区、乡村道路、郊区、弯道、夜间驾驶、混合交通。

E4：高概率。大于 40 小时/年，或几乎每次驾驶。例如直线道路、高速/城市通勤、正常天气、白天、高速公路巡航。

E 取决于操作场景暴露，不取决于故障概率。

## Controllability C

C0：一般可控。伤害可通过常规行为避免，或不会导致不可接受人身伤害风险。

C1：简单可控。大多数驾驶员通过直接常规反应可以避免伤害；提示明显，反应时间通常大于 5 秒。

C2：通常可控。部分驾驶员可通过及时适当反应避免伤害；反应时间通常 1-5 秒，需要较高注意力、空间或操作能力。

C3：难以控制或不可控。多数驾驶员或相关人员难以及时避免伤害；反应时间极短、小于 1 秒，或驾驶员不在控制回路中，或动作反直觉、空间不足。

C 判断必须说明：感知来源、反应时间、可用操作、道路空间、驾驶员是否在车上、其他交通参与者是否能避让。

## 功能族经验的使用方式

可以使用常识性功能族知识辅助判断，例如照明、制动、转向、驻车、动力等功能的物理作用、驾驶员可感知方式和风险对象。但功能族知识只能回答“这种故障在当前场景中通过什么机制影响风险”，不能直接替代场景推理。

使用功能族经验时必须完成三步：

1. 先从本行场景提取事实：操作状态、环境、速度、风险对象、附加条件、故障如何触发危险事件。
2. 再解释该功能的失效机制如何在这些事实下改变 S/E/C。
3. 最后检查解释中是否引入了本行没有的条件；若有，删除或改写。

不要写成固定规则，例如“某功能某故障通常为 C1/QM”。即便同一故障，在主动请求、未及时发现、驾驶员离车、高速近距离、低速起步、日间低风险等不同场景中，也可能得到不同的 E/C/S。

## ASIL 矩阵

本 skill 使用项目约定的后缀求和规则：

```text
score = S后缀 + E后缀 + C后缀
score <= 6 => QM
score == 7 => A
score == 8 => B
score == 9 => C
score >= 10 => D
```

示例：

- S3 + E4 + C3 = 10 => D
- S3 + E3 + C3 = 9 => C
- S2 + E4 + C2 = 8 => B
- S1 + E2 + C3 = 6 => QM

最终结果必须由脚本校验。


## 当前功能族物理知识
## 外部照明/前照灯

典型功能：

- 近光灯、远光灯、位置灯、自动灯光、故障提示、夜间开灯提示。

典型 malfunction：

- 近光灯无法点亮。
- 近光灯无法熄灭。
- 近光灯亮度过大，可能使其他道路使用者误判或眩目；外部照明的车辆级输出天然包含亮度/照度维度，即使需求只描述驱动开/关也要审查。
- 近光灯亮度过小，导致驾驶员视野降低。
- 应关闭时仍点亮或非预期点亮。
- 非预期关闭，尤其是夜间或隧道内由点亮状态变为熄灭状态。
- 近光灯点亮响应过晚。
- 近光灯熄灭响应过晚。
- 近光灯卡滞在关闭状态或卡滞在打开状态。
- 近光灯请求点亮却熄灭，或请求熄灭却点亮。
- 指示灯或提示错误，导致驾驶员误判外部照明状态。

典型整车级危害：

- 驾驶员视野丢失或者降低。
- 其他驾驶员/行人误判。
- 驾驶员获取错误的信息。
- 失去报警。

有效场景机制：

- 夜间、隧道、无路灯或弱照明道路，近光灯无法点亮，驾驶员前方视野不足，未及时发现行人、障碍物或前车。
- 夜间/隧道中近光灯非预期熄灭，驾驶员短时间内视野突然下降。
- 对向来车或前车因异常灯光状态误判本车位置或意图。
- 夜间开灯提示丢失只在驾驶员实际可能忘记开灯且外部照明不足时才有安全相关性。
- 近光灯非预期点亮在白天通常无危害，但在夜间、低照度、会车或复杂交通中可能影响其他道路使用者判断；不要只按白天工况判为无危害。

无效或弱场景：

- 白天正常天气仅近光灯不亮，通常安全影响较弱。
- 把“灯不亮导致违反法规”当作 HARA 人身伤害事件。
- 风险对象不在本车可照明/行驶路径内。
- 把近光灯非预期点亮默认等同于远光灯强眩目；若没有亮度过大或配光错误故障，不要自动推导成高速致命眩目。
- 近光灯无法点亮、亮度过小或卡滞熄灭在夜间刚起步/傍晚行驶中通常容易被驾驶员发现并采取措施，HARA 中 E/C 往往较低，常见场景可能为 QM。
- 近光灯点亮响应过晚若只是短暂延迟，通常不支持高 ASIL，除非场景中驾驶员在低照度下突然失去必要视野且无足够反应时间。
- 近光灯请求点亮却熄灭不能简单等同于非预期熄灭；它是有请求时的方向错误，驾驶员通常更容易发现。
- 近光灯请求熄灭却点亮通常不构成整车级危害；请求熄灭说明当前通常不需要近光灯，且驾驶员容易发现并再次操作。

# SEC 场景扎根审计

目标：审计 S/E/C 评级和解释是否由当前单条危险事件自然推出。审计者不能知道其他场景、用户期望答案或后处理规则。

## 审计原则

1. 场景事实优先：只使用本行 `道路类型`、`道路条件`、`环境条件`、`车辆状态`、`车速(km/h)`、`特殊要素`、`附加条件`、`危害事件`、`有风险的人员`、故障描述和整车危害。
2. 禁止补事实：SEC 解释不得引入场景中没有的时序、操作状态、环境、风险对象或驾驶员行为。若解释中出现“起步、傍晚、进入隧道、主动开灯、未及时发现、距离较近、行人横穿”等事实，必须能在本行场景字段或附加条件中找到对应依据。
3. 评级从场景推出：不要为了让 ASIL 变高或变低而改写 E/C/S 的理由。若现有评级不合理，应直接给出更合理的评级和理由。
4. 不按功能套模板：即使某个功能族有常见经验，也必须转化为当前场景的因果判断。不同系统、不同故障、不同场景都按同一审计方法处理。
5. 低风险也要可解释：QM 不是默认结论。若评级为 QM，必须说明是 S、E、C 中哪一项使风险降低，并且该降低原因来自场景事实。
6. 功能族知识只用于物理机制审查：可用于判断某故障是否真的能造成该危害、风险对象是否在影响路径内、可感知性是否合理；不能直接当成固定 E/C/S 结论。
7. 禁止假设性旁支：SEC 解释不得写“若存在/若无/可能还有”等本行未给出的替代条件。若需要某条件才能成立，应在本行场景中已有该条件，否则应删除该条件或把评级按现有场景重做。
8. 反模式优先：若当前 SEC 或危险事件违反 prompt 中的 HARA 评审反模式，`approved` 必须为 `false`，并在 `final_sec` 中按本行事实给出更合理的评级。不得因为原场景写得严重就顺着错误场景给高评级。

## S/E/C 审计步骤

### 1. 审计 E

判断 `E-解释` 是否描述当前操作场景的暴露频率，而不是故障概率或模板化场景。

合格的 E 解释应包含：

- 当前场景的核心暴露条件，例如道路类型、环境、车辆状态、速度范围、特殊要素或附加条件中的限定条件。
- 为什么这些条件组合是 E0/E1/E2/E3/E4。
- 若使用“低概率子场景”，必须指出子场景来自附加条件，而不是临时补充。

不合格示例类型：

- 场景为“夜间前向匀速”，E 解释却说“起步/傍晚开灯时”。
- 场景没有“行人横穿”，E 解释却以“学校区域行人横穿”作为暴露依据。
- E 解释把“故障发生概率低”当作暴露频率低。

### 2. 审计 S

判断 `可能的后果('S'的理由)` 是否说明风险对象、碰撞或伤害机制、速度/能量依据。

合格的 S 解释应包含：

- 谁受伤。
- 如何受伤，例如追尾、侧碰、撞击障碍物、撞击行人、车辆溜车撞击。
- 速度、相对速度、坡度/距离/能量等依据。

### 3. 审计 C

判断 `C-解释` 是否与场景中的可感知性、反应时间、可用操作和空间约束一致。

合格的 C 解释应包含：

- 异常如何被驾驶员或相关人员感知。
- 反应时间或发现时机是否充分。
- 可用操作，例如制动、转向、停车、切换模式、重新操作、避让。
- 场景是否限制这些操作，例如车速高、距离近、弯道、坡道、驾驶员离车、风险对象突然出现。

不合格示例类型：

- 场景写明“驾驶员未及时发现”，C 解释却说“驾驶员会立即发现”。
- 驾驶员离车场景按驾驶员可踩刹车处理。
- 高速近距离或弯道场景按充足反应时间处理，但没有说明可视距离、制动距离或道路空间。
- 场景没有对向来车，C 解释却写“若无对向来车可开启远光灯”。
- 场景是多车道隔离/封闭道路，解释却引入对向会车或正面碰撞机制。

### 4. 审计功能物理机制

使用当前 prompt 中的功能族物理知识检查：

- 故障是否是危险事件的必要原因。
- 危害路径是否符合该功能的物理作用范围。
- 风险对象是否真的位于该功能影响路径内。
- 可感知性和可控性是否符合该功能的典型表现。

如果功能族知识与当前 SEC 冲突，不要机械套用功能族结论；先回到本行场景，说明当前场景为什么适用或不适用该知识。

### 5. 审计危险事件是否被过度放大

SEC 复核不仅检查解释文字，还要检查当前危险事件是否把弱故障机制放大成不合理后果。若危险事件本身已经包含夸大的碰撞路径，但本行事实和功能物理知识不支持该路径，应在 S/C/E 中按合理机制评级，并在解释中使用合理机制，不继续放大。

常见审计问题：

- 弱感知干扰被写成高速正面碰撞，但没有亮度、配光、失控或车道侵入等支撑。
- “短暂延迟”被当成长期完全失效，但场景没有说明延迟窗口内已经进入不可避危险。
- 道路拓扑不支持风险对象，例如封闭快速路默认行人横穿。
- C 评级把“可能制动/减速/停车”完全忽略，或在距离、速度、空间明显受限时又写成完全容易控制。

## 修订要求

如果现有 SEC 通过审计，保持原评级和解释，只做必要的中文精炼。

如果现有 SEC 未通过审计，重新给出整组 `E-解释`、`暴露频率'E'`、`可能的后果('S'的理由)`、`Severity 'S'`、`C-解释`、`控制能力 'C'`。修订必须：

- 只使用本行事实。
- 不出现“评级校正”“修正”“按规则压低/提高”等后处理措辞。
- 不把审计发现写成输出解释。
- 不为了保留原 ASIL 而牺牲场景一致性。


## HARA 评审反模式
# HARA 评审规则

## Blocking 规则

以下问题必须修复后才能交付：

- 字段缺失、sheet 名错误、枚举不在允许值中。
- `derive_mf.子功能` 偏离源文档功能清单，或把功能内部逻辑误拆成独立功能。
- sheet1 故障描述不是功能行为偏离，而是根因、场景、危害事件或安全机制。
- MF、Milf、HARA、SG 编号重复或不符合规则。
- sheet2 故障描述不能追溯到 sheet1。
- sheet2 把当前故障偷换成另一个引导词或另一个故障，例如把 `EPB夹紧力过大` 当成 `EPB非预期拉起`。
- HARA 故障/整车危害不能追溯到 sheet2。
- ASIL 与 S/E/C 矩阵不一致。
- 非 QM HARA 行缺少安全目标、安全状态或 FTTI。
- QM HARA 行填写了安全目标、安全状态或 FTTI。
- 危险事件中故障不是必要原因。
- 车辆状态与车速明显矛盾，例如驻车 + 高速。
- 场景超出功能操作域且没有合理越域故障解释。
- 风险对象不在车辆危险路径内。
- 道路类型与风险对象/特殊要素互相冲突，例如高速公路默认出现行人横穿、骑行者或城市路口。
- 非 QM 条目没有进入 `sg_sum`。
- 非 `无危害` 故障的有效 HARA 场景少于 3 条且没有明确排除理由。

## Warning 规则

以下问题应尽量修复，若保留需要在评审报告说明：

- 使用 `ALL` 或 `不涉及` 过多，导致场景不可评审。
- 只通过改变天气、道路名或车速生成重复场景。
- 同一故障场景没有覆盖常见高暴露、最高合理严重度、低可控性/边界工况。
- sheet1 漏掉明显的作用量过大/过小，例如亮度、夹紧力、制动力、扭矩、角度、速度或持续时间。
- sheet1 把驾驶员主动离散请求误写成过早，而不是非预期激活。
- sheet1 的功能丧失按多个前置条件重复列出冗余故障，而不是概括为一个功能级偏离。
- sheet1 的卡滞没有写明卡滞状态。
- sheet1 的方向错误没有写明请求方向和实际执行方向相反。
- `E-解释` 没有说明频率或使用占比。
- `可能的后果('S'的理由)` 没有说明伤害对象和伤害机制。
- `C-解释` 没有说明感知、反应时间、空间或驾驶员是否在车上。
- 安全目标包含实现词：传感器、算法、报文、CAN、驱动、诊断、DTC、软件、硬件、冗余。
- FTTI 只有数字但没有基于场景或危害的解释来源。

## 反模式

### AP-001 天气替代因果

雨、雪、雾、夜间、低附着不能单独造成 item 故障导致的危害。它们只能作为暴露或严重度/可控性的影响因素。

### AP-002 驻车高车速

车辆状态为驻车、停车熄火、停车未熄火时，车速通常应为静止状态或低速蠕行。若写高速，必须是非预期移动或功能越域并有解释。

反向也成立：车辆状态为前向匀速、前向加速、转弯、变道、超车等行驶状态时，车速不能写静止状态。

### AP-003 风险对象错位

危险事件中的受伤人员必须位于危险路径内。前方障碍物不能支持“对向车辆乘员受伤”，后方来车不能支持“行人被前照灯照明不足撞击”。

### AP-004 普通交通风险冒充功能安全风险

如果没有 item 故障，同样场景仍会发生同样危险事件，则不是有效 HARA 危险事件。

### AP-005 安全目标设计化

安全目标不能写成具体实现机制。设计机制应出现在功能安全概念或技术安全需求中，不应作为 SG。

### AP-006 过大故障偷换成非预期激活

作用量过大只描述请求动作发生时输出过高，不能为了构造危害加入“非预期动作”。如果当前故障只会造成零件磨损、寿命下降、能耗或舒适性问题，应映射为 `无危害`。

### AP-007 近光灯按远光灯夸大

近光灯异常不能默认等同于远光灯强眩目。视野不足、误判或短暂延迟类场景要检查驾驶员是否通常能提前发现、是否有心理预期和可用应对措施；常见起步/傍晚场景往往是 QM。

### AP-008 EPB驻车溜车直接按最高E/C

EPB拉起失败、夹紧力不足、响应过晚、请求拉起却释放等驻车溜车场景，必须有坡度、外力、驱动蠕行或已有保持力释放作为纵向力来源。驾驶员仍在车上时通常可通过脚刹、P挡或重新操作补救，不应轻易给 C3；暴露率应针对坡道/行人/外力组合场景评估，不能把“每次停车”直接等同于该危险场景的 E4。

### AP-009 道路类型与风险对象冲突

高速公路、城市快速路等封闭或快速通行道路，不应默认存在行人横穿、骑行者、红绿灯路口或环岛路口。若确实需要弱势交通参与者，必须说明施工区域、事故场景、服务区、收费站、应急车道、路肩、救援或养护等特殊前提。否则应改用更符合道路环境的危险对象，例如前方停驶车辆、慢行车辆、遗落货物、障碍物、护栏或前车乘员。

## 最终评审提问

逐行检查：

1. 故障描述是否来自正常功能偏离？
2. 整车危害是否是车辆级状态？
3. 车辆状态、车速、驾驶员是否在车上是否运动学一致？
4. 道路类型、道路条件、特殊要素、风险对象是否属于同一种道路拓扑？
5. 环境条件是否支撑危害事件中的夜间、日间、雨雪、湿滑、低能见度等描述？
6. 风险对象是否真的位于车辆危险路径内？
7. 危害事件是否和当前故障、当前整车危害属于同一风险机理？
8. 移除故障后，危险事件是否消失或显著降低？
9. S/E/C 理由是否只使用当前场景事实？
10. ASIL 是否由工具计算？
11. SG 是否覆盖最高 ASIL 事件？
