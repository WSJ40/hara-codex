# HARA 工作流与上下文管理

## 目标

本 skill 用于从功能需求文档生成可评审的 HARA Excel。质量优先级：

1. 不漏掉需求文档中的功能和安全相关故障。
2. 不接受故障不是必要原因的危险事件。
3. 不接受物理、车辆状态、速度、风险对象位置互相矛盾的场景。
4. 不让模型手算或猜测 ASIL。
5. 非 QM 风险必须可追溯到安全目标。

## 上下文隔离

不要把“分阶段”理解成同一 Claude 会话里逐步读文件。那样总上下文仍会累积，SEC 阶段仍可能带着 malfunction 推理、候选场景草稿和被拒绝场景。必须优先使用 `scripts/claude_hara_pipeline.py`，每个阶段启动独立 Claude 调用，并用 JSON 文件作为唯一记忆。

最小隔离边界：

1. `item_functions`：只读源文档抽取、功能清单、HARA 核心定义。
2. `derive_mf:<function>`：只读 item 摘要、当前功能片段、引导词定义、相关功能族模式。
3. `map_hazard:<function>`：只读当前功能的 malfunction 和整车危害枚举。
4. `scenario_generate:<fault>`：只读一个故障、一个整车危害、当前功能片段、场景枚举、场景规则；禁止读取 SEC 指南。
5. `scenario_review:<fault>`：只读候选场景和 review rules；不能读取生成器的自由推理。
6. `sec_rate:<scenario>`：只读一个已接受场景和 SEC 指南；不能读取被拒绝候选场景或早期引导词推理。
7. `safety_goal:<fault-group>`：只读非 QM rated scenarios、HARA 核心定义、安全目标规则。
8. `final_review`：只读最终 JSON 和 review rules，运行工具校验。

多功能需求文档必须分片处理：

1. 读取功能清单和目录。
2. 对每个功能，只读取该功能小节、共用 item 边界、必要的接口/状态描述。
3. 生成该功能的 malfunction 和 hazard 后立刻写入 JSON 草稿。
4. 进入下一个功能前，总结已确认的 item 级事实，不携带上一功能的细节。
5. 场景生成按“一个故障/危害一轮”处理，最多同时看同一故障的候选场景。
6. SEC 评级时只读单条已接受场景和 `sec-guide.md`，不要重新读取整份需求文档、malfunction 草稿或场景生成上下文。

## 阶段质量门

### Item 定义

- 系统边界必须来自文档或明确假设。
- 操作模式、电源档位、车辆状态必须来自文档或保守假设。
- 不清楚时写入假设，不要隐式补全。

### Malfunction

- 故障必须是功能行为偏离，不是零件失效根因。
- `derive_mf.子功能` 必须优先使用源文档功能清单中的功能名；不要把功能内部逻辑、告警、状态上报自动拆成独立功能，除非源文档功能清单明确列出。
- 每个故障必须能回答：正常功能是什么、偏离是什么、车辆表现是什么。
- 先读取 `sheet1-malfunction-guide.md`，建立当前功能的行为模型，再逐个判断 8 个引导词是否存在对应功能偏离。
- 只要功能故障逻辑存在，就写入 sheet1；不要提前用安全相关性、ASIL 预期或场景可想象程度筛掉故障。
- 控制故障粒度：sheet1 写功能级偏离，同一偏离不要按前置条件重复拆分；不同作用量、状态方向或相反执行效果要分清。
- 无安全影响的故障可以保留，但后续应映射为 `无危害`。

### 整车级危害

- 危害必须是车辆级状态，例如“非预期的减速”“驾驶员视野丢失或者降低”。
- 组件级、信号级、DTC、报文错误不是整车级危害。

### 场景

- 场景必须在功能可发生的操作域内；非预期激活可以发生在预期操作域外，但要说明为什么。
- 每个安全相关故障先生成 4-8 条候选场景，再经评审去重和过滤；最终 HARA 中每个非 `无危害` 故障通常不少于 3 条场景。
- 少于 3 条时，必须在 `附加条件` 或项目假设中说明为什么该故障操作域非常窄、其他候选场景被合理排除。
- 风险对象必须位于危险路径内。
- 天气、低附着、夜间、拥堵只能作为放大因素，不能替代故障因果。
- 类似场景只保留 ASIL 最高或机制最有代表性的一条。

### S/E/C

- S 只看伤害严重性，不看发生概率或可控性。
- E 只看该操作场景暴露频率，不看伤害严重性。
- C 只看典型人员是否能避免伤害，不用 ASIL 反推。
- 理由必须引用场景事实。

### Safety Goal

- 安全目标是车辆级安全意图，不是技术方案。
- 可合并相同安全意图的危险事件，但继承最高 ASIL。
- 每个非 QM 最高风险故障都要在 `sg_sum` 中覆盖。

## 迭代策略

如果脚本校验失败，先修复确定性问题：字段、枚举、ID、ASIL、追溯。再修复工程质量问题：场景因果、SEC 理由、安全目标。最后重新导出。


# 功能族模式

只读取与当前系统相关的段落。下面是生成 malfunction、hazard、scenario 的参考，不是强制模板。

## 外部照明/前照灯

典型功能：

- 近光灯、远光灯、位置灯、自动灯光、故障提示、夜间开灯提示。

典型 malfunction：

- 近光灯无法点亮。
- 近光灯无法熄灭。
- 近光灯亮度过大，可能使其他道路使用者误判或眩目；外部照明的车辆级输出天然包含亮度/照度维度，即使需求只描述驱动开/关也要审查。
- 近光灯亮度过小，导致驾驶员视野降低。
- 应关闭时仍点亮或非预期点亮。
- 非预期关闭，尤其是夜间或隧道内由点亮状态变为熄灭状态。
- 近光灯点亮响应过晚。
- 近光灯熄灭响应过晚。
- 近光灯卡滞在关闭状态或卡滞在打开状态。
- 近光灯请求点亮却熄灭，或请求熄灭却点亮。
- 指示灯或提示错误，导致驾驶员误判外部照明状态。

典型整车级危害：

- 驾驶员视野丢失或者降低。
- 其他驾驶员/行人误判。
- 驾驶员获取错误的信息。
- 失去报警。

有效场景机制：

- 夜间、隧道、无路灯或弱照明道路，近光灯无法点亮，驾驶员前方视野不足，未及时发现行人、障碍物或前车。
- 夜间/隧道中近光灯非预期熄灭，驾驶员短时间内视野突然下降。
- 对向来车或前车因异常灯光状态误判本车位置或意图。
- 夜间开灯提示丢失只在驾驶员实际可能忘记开灯且外部照明不足时才有安全相关性。
- 近光灯非预期点亮在白天通常无危害，但在夜间、低照度、会车或复杂交通中可能影响其他道路使用者判断；不要只按白天工况判为无危害。

无效或弱场景：

- 白天正常天气仅近光灯不亮，通常安全影响较弱。
- 把“灯不亮导致违反法规”当作 HARA 人身伤害事件。
- 风险对象不在本车可照明/行驶路径内。
- 把近光灯非预期点亮默认等同于远光灯强眩目；若没有亮度过大或配光错误故障，不要自动推导成高速致命眩目。
- 近光灯无法点亮、亮度过小或卡滞熄灭在夜间刚起步/傍晚行驶中通常容易被驾驶员发现并采取措施，HARA 中 E/C 往往较低，常见场景可能为 QM。
- 近光灯点亮响应过晚若只是短暂延迟，通常不支持高 ASIL，除非场景中驾驶员在低照度下突然失去必要视野且无足够反应时间。
- 近光灯请求点亮却熄灭不能简单等同于非预期熄灭；它是有请求时的方向错误，驾驶员通常更容易发现。
- 近光灯请求熄灭却点亮通常不构成整车级危害；请求熄灭说明当前通常不需要近光灯，且驾驶员容易发现并再次操作。

## 电子驻车制动 EPB/驻车保持

典型功能：

- 静态开关拉起、释放、自动驻车、坡道驻车、休眠唤醒后拉起。

典型 malfunction：

- EPB无法拉起。
- EPB拉起夹紧力过大。
- EPB拉起夹紧力不足。
- EPB拉起响应过晚。
- EPB非预期拉起；未请求、行驶中、不满足允许条件等是同一本质故障的场景/操作域变体，sheet1 不重复拆分。
- EPB卡滞在释放状态导致无法拉起。
- EPB卡滞在拉起状态导致无法释放。
- EPB请求拉起却执行释放，或请求释放却执行拉起。
- 驻车状态反馈错误，驾驶员误判车辆已驻稳。

典型整车级危害：

- 非预期的纵向移动。
- 非预期的减速。
- 无法纵向移动。
- 驾驶员获取错误的信息。

有效场景机制：

- 坡道驻车或停车未熄火时，EPB 请求拉起失败，车辆因坡度、驱动蠕行或外力发生非预期移动，撞击行人、后车、障碍物。
- 静止/低速时 EPB 非预期拉起，引起车辆突然减速或后车追尾。
- 行驶中 EPB 非预期拉起属于低可控性高风险场景；若当前功能只允许静态拉起，必须说明非预期激活如何越过操作域。
- EPB 夹紧状态反馈错误可能导致驾驶员离车，车辆随后移动，驾驶员不在车上时 C 往往更差。

无效或弱场景：

- EPB夹紧力过大在静态拉起功能中通常只影响制动器、电机、卡钳或机构寿命，不应映射为非预期减速；非预期减速属于 EPB 非预期拉起。
- “雨雪导致车辆溜坡”但没有坡度、驱动扭矩、外力或制动力不足作为纵向力来源。
- 车辆状态为驻车却给出 60 km/h 以上车速。
- 静态开关拉起功能的故障被直接扩展为高速巡航故障，除非故障模式明确为非预期激活且解释越域。
- 把 Released 状态请求拉起失败、休眠唤醒失败等不同前置条件重复写成多个功能丧失故障；sheet1 应概括为“驻车时 EPB 无法拉起”，前置条件在 sheet2/HARA 展开。
- 驾驶员主动拉起开关通常没有“过早”故障；请求前拉起属于非预期激活。

## 制动系统

典型危害包括失去减速、制动能力降低、非预期减速。核心场景关注前车距离、速度、路面附着、弯道/直道、后车距离和驾驶员可用反应时间。

## 转向系统

典型危害包括失去横向运动、非预期横摆、方向错误。核心场景关注弯道、变道、车道保持、对向车辆、道路边界和驾驶员施力能力。

## 动力/驱动系统

典型危害包括非预期加速、失去加速、相反方向纵向移动。核心场景关注起步、超车、路口、拥堵跟车、坡道、倒车和驾驶员可通过制动/换挡控制的能力。

## HMI/提示/报警

HMI 本身通常不是直接伤害源。只有当错误信息、误报警、漏报警会改变驾驶员决策并导致车辆进入危险状态时，才形成危险事件。必须说明驾驶员基于错误信息采取了什么动作。


# HARA 评审规则

## Blocking 规则

以下问题必须修复后才能交付：

- 字段缺失、sheet 名错误、枚举不在允许值中。
- `derive_mf.子功能` 偏离源文档功能清单，或把功能内部逻辑误拆成独立功能。
- sheet1 故障描述不是功能行为偏离，而是根因、场景、危害事件或安全机制。
- MF、Milf、HARA、SG 编号重复或不符合规则。
- sheet2 故障描述不能追溯到 sheet1。
- sheet2 把当前故障偷换成另一个引导词或另一个故障，例如把 `EPB夹紧力过大` 当成 `EPB非预期拉起`。
- HARA 故障/整车危害不能追溯到 sheet2。
- ASIL 与 S/E/C 矩阵不一致。
- 非 QM HARA 行缺少安全目标、安全状态或 FTTI。
- QM HARA 行填写了安全目标、安全状态或 FTTI。
- 危险事件中故障不是必要原因。
- 车辆状态与车速明显矛盾，例如驻车 + 高速。
- 场景超出功能操作域且没有合理越域故障解释。
- 风险对象不在车辆危险路径内。
- 非 QM 条目没有进入 `sg_sum`。
- 非 `无危害` 故障的有效 HARA 场景少于 3 条且没有明确排除理由。

## Warning 规则

以下问题应尽量修复，若保留需要在评审报告说明：

- 使用 `ALL` 或 `不涉及` 过多，导致场景不可评审。
- 只通过改变天气、道路名或车速生成重复场景。
- 同一故障场景没有覆盖常见高暴露、最高合理严重度、低可控性/边界工况。
- sheet1 漏掉明显的作用量过大/过小，例如亮度、夹紧力、制动力、扭矩、角度、速度或持续时间。
- sheet1 把驾驶员主动离散请求误写成过早，而不是非预期激活。
- sheet1 的功能丧失按多个前置条件重复列出冗余故障，而不是概括为一个功能级偏离。
- sheet1 的卡滞没有写明卡滞状态。
- sheet1 的方向错误没有写明请求方向和实际执行方向相反。
- `E-解释` 没有说明频率或使用占比。
- `可能的后果('S'的理由)` 没有说明伤害对象和伤害机制。
- `C-解释` 没有说明感知、反应时间、空间或驾驶员是否在车上。
- 安全目标包含实现词：传感器、算法、报文、CAN、驱动、诊断、DTC、软件、硬件、冗余。
- FTTI 只有数字但没有基于场景或危害的解释来源。

## 反模式

### AP-001 天气替代因果

雨、雪、雾、夜间、低附着不能单独造成 item 故障导致的危害。它们只能作为暴露或严重度/可控性的影响因素。

### AP-002 驻车高车速

车辆状态为驻车、停车熄火、停车未熄火时，车速通常应为静止状态或低速蠕行。若写高速，必须是非预期移动或功能越域并有解释。

### AP-003 风险对象错位

危险事件中的受伤人员必须位于危险路径内。前方障碍物不能支持“对向车辆乘员受伤”，后方来车不能支持“行人被前照灯照明不足撞击”。

### AP-004 普通交通风险冒充功能安全风险

如果没有 item 故障，同样场景仍会发生同样危险事件，则不是有效 HARA 危险事件。

### AP-005 安全目标设计化

安全目标不能写成具体实现机制。设计机制应出现在功能安全概念或技术安全需求中，不应作为 SG。

### AP-006 过大故障偷换成非预期激活

作用量过大只描述请求动作发生时输出过高，不能为了构造危害加入“非预期动作”。如果当前故障只会造成零件磨损、寿命下降、能耗或舒适性问题，应映射为 `无危害`。

### AP-007 近光灯按远光灯夸大

近光灯异常不能默认等同于远光灯强眩目。视野不足、误判或短暂延迟类场景要检查驾驶员是否通常能提前发现、是否有心理预期和可用应对措施；常见起步/傍晚场景往往是 QM。

## 最终评审提问

逐行检查：

1. 故障描述是否来自正常功能偏离？
2. 整车危害是否是车辆级状态？
3. 场景是否在功能可发生的范围内？
4. 移除故障后，危险事件是否消失？
5. 场景字段之间是否互相一致？
6. S/E/C 理由是否只使用当前场景事实？
7. ASIL 是否由工具计算？
8. SG 是否覆盖最高 ASIL 事件？

11: 1.2.1 系统工作原理
12: 
13: 在任意电源档位下，通过切换组合开关上的灯光挡位来亮灭近光灯。前车身控制器作为信号采集和处理的主控，接收组合开关发送的灯光档位信号来控制近光灯的亮灭。若为美版车，则仪表显示系统还需要同步点亮灯光总开关指示灯。
14: 
15: 1.2.2 功能逻辑
16: 
17: 1.2.2.1 近光灯亮灭逻辑
18: 
19: ● 输出控制描述：
20: 
21: （1）控制近光灯熄灭：近光灯状态为熄灭，断开近光灯驱动，发送“近光灯”报文（0xXXX，X.X-X.X,0x2）。
22: 
23: （2）控制近光灯点亮：近光灯状态为点亮，输出近光灯驱动，发送“近光灯”报文（0xXXX，X.X-X.X,0x1）。
24: 
25: ● 逻辑描述：
26: 
27: （1）模块得常电需要实时记忆近光灯状态（点亮、熄灭），断电不记忆。
28: 
29: （2）满足下面任意条件，控制近光灯点亮：
30: 
31: （a）近光灯档：接收到1帧“灯光旋钮当前机械结构处于的档位”报文（0x133，4.3-4.5，0x4），或掉线时判断为近光灯档。
32: 
33: （b）AUTO档：根据自动灯光逻辑，判断为近光灯点亮时。
34: 
35: （3）满足下面任意条件，控制近光灯熄灭：
36: 
37: （a）非近光灯/AUTO档：接收到1帧“灯光旋钮当前机械结构处于的档位”报文（0x133，4.3-4.5，0x1/0x3）。
38: 
39: （b）AUTO档：根据自动灯光逻辑，判断为近光灯熄灭时。
40: 
41: （4）掉线处理：连续5s未接收到“灯光旋钮当前机械结构处于的档位”报文（0x133，4.3-4.5）时，认为掉线，执行如下：
42: 
43: （a）ON档电下掉线，若此时采集到近光灯开启/关闭信号为低电平，认为此时处于近光灯档。
44: 
45: （b）ON档电下掉线，若此时采集到近光灯开启/关闭信号为非低电平，默认保持上一状态。
46: 
47: （c）OFF档电下掉线，默认保持上一状态。
48: 
49: （5）整车处于防盗设定状态/防盗状态时，不处理灯光开关的变化（判断“灯光旋钮当前机械结构处于的档位”报文）。
50: 
51: （6）模块上常电默认灯光旋钮的左灯光档当前处于OFF档，近光灯状态为熄灭，断开近光灯驱动，发送“近光灯”报文（0xXXX，X.X-X.X,0x2）。
52: 
53: （7）模块在不掉常电下出现自复位时，近光灯状态需要保持自复位前的状态，然后再按正常逻辑执行。
54: 
55: 1.2.2.2 灯光总开关指示灯逻辑
56: 
57: 若为美版出口车，在控制近光灯亮灭时，需控制仪表显示系统的灯光总开关指示灯同步亮灭。非美版出口车，无此指示灯。
58: 
59: 仪表自身会要求对美版出口车做区分。
60: 
61: 1.2.2.3 夜间行车语音提示功能
62: 
63: （1）同时满足下面所有条件，按照事件周期格式先以短周期发送三帧“夜间行车开灯语音提示”报文（0xXXX，X.X，0x1），然后再发送“夜间行车开灯语音提示”报文（0xXXX，X.X，0x0）：
64: 
65: （a）电源档位为ON档。
66: 
67: （b）灯光开关处于OFF档或小灯档：接收到1帧组合开关发送的“灯光旋钮当前机械结构处于的档位”报文（0x133，4.3-4.5，0x1/0x3）。
68: 
69: （c）车速≥5km/h：接收到1帧“车速信号”报文（0x121，1.0-2.3，车速≥5）。
70: 
71: （d）外界光强档位为5档：发送“光照强度值”CAN报文（0xXXX，X.X-X.X，0x5）时。
72: 
73: （2）不满足上面任一条件（（a）-（d）），发送“夜间行车开灯语音提示”报文（0xXXX，X.X，0x0）。
74: 
75: （3）掉线处理：连续5s未接收到“灯光旋钮当前机械结构处于的档位”报文时，认为组合开关掉线，报文默认保持上一状态；连续10帧未接收到“车速信号”报文时，认为ESP掉线，报文默认保持上一状态。
76: 
77: （4）模块上常电默认发送“夜间行车开灯语音提示”报文（0xXXX，X.X，0x0）。

80: 
81: 当前域需要输出近光灯驱动时，需要对近光灯做故障检测。
82: 
83: （1）在近光灯工作时，满足下面任一条件，认为近光灯故障，发送“近光灯故障状态”报文（0xXXX，X.X-X.X，0x2）：
84: 
85: （a）诊断当前故障有“近光灯驱动电路对地短路故障”。
86: 
87: （b）诊断当前故障有“近光灯驱动电路断路故障”。
88: 
89: （2）在近光灯工作时，满足下面所有条件，认为近光灯正常，发送“近光灯故障状态”报文（0xXXX，X.X-X.X，0x1）：
90: 
91: （a）诊断当前故障无“近光灯驱动电路对地短路故障”。
92: 
93: （b）诊断当前故障无“近光灯驱动电路断路故障”。
94: 
95: （3）在近光灯不工作时，默认近光灯正常。
96: 
97: （4）模块得常电或者休眠唤醒时，默认近光灯正常。
