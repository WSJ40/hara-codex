# HARA 核心知识

## HARA 边界

HARA 是 ISO 26262 概念阶段的危害分析与风险评估活动，用于识别由 item 功能异常引起的车辆级危险事件，评价 S/E/C 并得到 ASIL 或 QM，随后形成车辆级安全目标。

HARA 不做：

- 组件 FMEA。
- 根因诊断。
- 软件详细设计。
- 安全机制设计。

## Item Definition

Item 定义应说明被分析系统的功能边界、车辆级预期行为、操作模式、外部接口、驾驶员/用户角色、假设和排除项。需求文档没有写清楚的信息必须作为假设保留。

## Malfunction

Malfunction 是预期功能的行为偏离。描述要从车辆/用户可观察行为出发，不要写根因。

常用引导词：

- 功能丧失：应提供功能时完全不提供或无法保持。
- 过大：输出、强度、持续时间、作用量超过预期。
- 过早：功能在合理时机之前发生。
- 过小：输出、强度、持续时间、作用量不足。
- 过晚：功能响应延迟或结束延迟。
- 非预期激活：没有请求或不满足条件时激活，或应保持关闭时激活。
- 卡滞：状态、输出、执行器或显示保持在某一状态无法改变。
- 方向错误：车辆运动、执行方向、显示指示与预期相反。

引导词不是填空游戏，也不是安全风险筛选器。sheet1 阶段先建立功能行为模型，再判断每个引导词对应的偏离是否逻辑存在。只要故障存在，就应进入 sheet1；是否产生车辆级危害和 ASIL 等级在后续阶段判断。

功能行为模型至少包含触发源、允许条件、执行动作、目标状态、物理作用量、时间关系、相反方向/状态和反馈提示。基于这些维度判断功能丧失、过大、过小、过早、过晚、非预期激活、卡滞和方向错误是否存在。

## Vehicle-Level Hazard

整车级危害是车辆层面的危险状态，不是零部件故障。优先使用枚举：

- 纵向运动：非预期加速、失去加速、非预期减速、失去减速、制动能力降低、非预期纵向移动、无法纵向移动、相反方向纵向移动、俯仰/滚动。
- 侧向运动：非预期侧倾、非预期横摆、失去横向运动、侧倾/滚动。
- 垂向运动：非预期垂直运动、横摆/滚动。
- 特殊危害：热事件、电击、视野降低、错误信息、失去报警、其他道路使用者误判、需要保护时失去保护、无危害。

## Hazardous Event

危险事件 = 整车级危害 + 操作场景 + 风险对象/人员 + 可能伤害机制。

有效危险事件必须通过反事实检查：如果移除 item 故障，事件不应以同样方式发生。

## Safety Goal

安全目标是车辆级安全意图，避免或控制危险事件的不可接受风险。它应描述需要防止的危险状态，例如：

- 防止车辆在驻车状态下发生非预期移动。
- 防止夜间行驶时前照明能力丧失导致驾驶员视野不足。

安全目标不应写成：

- 增加传感器冗余。
- 优化算法。
- 检测 CAN 报文丢失。
- 驱动电路诊断。


# Sheet1 Malfunction 思维模型

本指南只用于 `derive_mf` 阶段。sheet1 的任务不是筛选安全风险，而是把当前功能所有逻辑存在的功能故障写出来；安全相关性、整车危害和场景严重性放到后续 sheet 判断。

## 核心判断

只要某个功能故障在当前功能的正常行为模型中逻辑存在，就应写入 sheet1。不要因为暂时想不到危害、ASIL 可能较低、源文档没有明示该故障，就把它从 sheet1 删除。

sheet1 只回答：

```text
这个功能本来应该怎样工作？
它可能以哪些方式偏离这个正常行为？
这些偏离分别属于哪个引导词？
```

sheet1 不回答：

```text
会不会造成事故？
事故发生在哪个道路场景？
ASIL 是多少？
安全目标是什么？
根因是哪个电路/报文/软件？
```

## 第一步：建立功能行为模型

从源文档提取当前功能的行为元素。不要只看功能名称，要看输入、条件、输出和状态。

- 触发源：驾驶员请求、自动条件、远程命令、系统状态变化。
- 允许条件：电源档位、车速范围、系统状态、互锁条件、网络/休眠状态。
- 执行动作：点亮、熄灭、拉起、释放、制动、加速、转向、锁止、提示。
- 目标状态：打开/关闭、Applied/Released、锁止/解锁、启用/停用。
- 物理作用量：亮度、夹紧力、制动力、扭矩、角度、速度、压力、温度、持续时间。
- 时间关系：响应时间、保持时间、自动触发时机、阶段顺序。
- 方向/极性：打开对关闭、拉起对释放、左对右、加速对减速、前进对后退。
- 反馈/提示：状态显示、报警、声音、指示灯；只有它们影响用户决策时才继续进入危害分析。

## 第二步：逐引导词做“存在性判断”

对 8 个引导词逐个回答：当前功能行为模型里是否存在这个偏离维度？存在就写；不存在才留空。

同一个异常行为只放入最贴切的一个引导词，避免跨列重复。优先选择更具体的引导词：

```text
卡在某状态无法改变 -> 卡滞
请求 A 却执行 B -> 方向错误
没有请求或不满足条件却动作 -> 非预期激活
请求后响应慢 -> 过晚
作用量太大/太小 -> 过大/过小
应提供主要功能但没有提供 -> 功能丧失
```

例如，应熄灭时一直点亮，若本质是状态无法改变，应归入“卡滞在点亮状态”，不要同时作为“功能丧失”重复写入。

### 功能丧失

判断问题：在应提供功能时，功能是否可能完全不提供或不能保持？

写功能级偏离，不写前置条件组合。

功能丧失优先表示“应提供主要功能但没有提供”。如果异常本质是某个状态无法改变、无法停止、无法退出，应优先归入 `卡滞`、`过晚` 或 `方向错误`，不要重复写在功能丧失里。

好：
- `驻车时 EPB 无法拉起`
- `近光灯无法按请求点亮`

差：
- `车辆静止且 EPB 处于 Released 状态时无法拉起；整车 OFF 或网络休眠状态下唤醒失败无法拉起`
- `近光灯应熄灭时无法熄灭`，如果已经写了 `近光灯卡滞在点亮状态`

前者是同一故障在不同条件下的场景化展开，应留到 sheet2/HARA；后者是与卡滞重复。

### 过大

判断问题：功能输出、作用量、强度、持续时间或效果是否可能超过预期？

存在物理作用量就要思考过大，即使功能表面上是开/关、拉起/释放。开关动作背后常有亮度、力、扭矩、制动力、保持时间等作用量。

示例：
- 照明功能可能有亮度/照度过大。
- 驻车/制动功能可能有夹紧力或制动力过大。
- 转向功能可能有转角或助力过大。

### 过小

判断问题：功能输出、作用量、强度、持续时间或效果是否可能不足？

示例：
- 照明功能可能有亮度/照度过小。
- 驻车/制动功能可能有夹紧力或制动力不足。
- 动力功能可能有扭矩过小。

### 过早

判断问题：功能是否有独立于用户即时请求的预期触发时刻、自动时序或阶段顺序？如果有，是否可能在正确时机之前发生？

主动离散请求本身通常没有“过早”。如果功能在没有请求前发生，通常归入 `非预期激活`，不是 `过早`。

只有当前功能自身拥有自动触发时机或阶段顺序时，才写过早。如果文档只是说“根据某上游自动逻辑判断后执行”，而该上游逻辑不是当前功能清单中的功能，则上游判断过早不要写入当前功能 sheet1；当前功能只分析收到请求/条件后的执行偏离。

好：
- 自动灯光在环境仍明亮时过早点亮。
- 自动释放在允许条件满足前释放。

通常不写：
- 驾驶员按开关打开/关闭。
- 驾驶员拉起 EPB 开关。

### 过晚

判断问题：请求或触发条件已经成立后，功能是否可能响应延迟、结束延迟或状态建立延迟？

示例：
- 点亮/熄灭响应延迟。
- 拉起/释放响应延迟。
- 制动力建立延迟。

### 非预期激活

判断问题：没有请求、不满足允许条件、或应保持当前状态时，功能是否可能发生状态改变或动作输出？

对于双向状态功能，要思考两个方向：

- 不该打开时打开。
- 不该关闭时关闭。
- 不该拉起时拉起。
- 不该释放时释放。

这些可以在同一引导词单元格内用同一 MF 编号分号分隔，后续 sheet2 再拆分。

### 卡滞

判断问题：功能状态、输出或执行器是否可能保持在某个状态，无法按请求改变？

必须写明卡滞状态，不能只写“状态卡滞”。

示例：
- 卡滞在打开状态。
- 卡滞在关闭状态。
- 卡滞在拉起状态。
- 卡滞在释放状态。

### 方向错误

判断问题：功能是否存在相反方向、相反状态或相反效果？如果存在，请求 A 时是否可能执行 B？

示例：
- 请求打开却关闭。
- 请求关闭却打开。
- 请求拉起却释放。
- 请求左转却向右转。
- 请求加速却减速。

## 第三步：控制 sheet1 粒度

sheet1 描述“故障是什么”，不描述“在什么场景发生”。使用以下粒度：

- 同一功能偏离，不因电源档位、状态、休眠唤醒、道路、天气、人群、车速不同而拆成多个 sheet1 故障。
- 不同偏离方向要拆开或在同一引导词下列出，例如非预期打开和非预期关闭。
- 不同物理作用量要拆开，例如夹紧力过大和夹紧力不足。
- 根因、检测机制、报文、诊断、电路、软件策略不要放入 sheet1。
- 如果某故障存在但后续没有安全影响，仍可在 sheet1 保留；sheet2 可映射为 `无危害` 或在备注说明。

## 输出前自检

1. 是否先建立了触发源、条件、动作、状态、作用量、时序、方向的功能行为模型？
2. 每个非空故障是否都是功能行为偏离？
3. 是否把“是否安全相关”提前用于删减 sheet1？如果是，恢复该故障。
4. 是否漏掉作用量的过大/过小？
5. 是否把无请求发生误写成过早？
6. 是否把同一偏离按前置条件重复写成多个故障？
7. 卡滞是否写明状态？
8. 方向错误是否写明请求和实际相反？


# 功能族模式

只读取与当前系统相关的段落。下面是生成 malfunction、hazard、scenario 的参考，不是强制模板。

## 外部照明/前照灯

典型功能：

- 近光灯、远光灯、位置灯、自动灯光、故障提示、夜间开灯提示。

典型 malfunction：

- 应点亮时不亮或亮度不足。
- 亮度过大，可能使其他道路使用者误判或眩目；只有源功能存在亮度/驱动强度维度时才使用。
- 亮度过小，导致驾驶员视野降低。
- 应关闭时仍点亮或非预期点亮。
- 非预期关闭，尤其是夜间或隧道内由点亮状态变为熄灭状态。
- 点亮/熄灭延迟。
- 灯光卡滞在关闭状态或卡滞在打开状态。
- 请求点亮却熄灭，或请求熄灭却点亮。
- 指示灯或提示错误，导致驾驶员误判外部照明状态。

典型整车级危害：

- 驾驶员视野丢失或者降低。
- 其他驾驶员/行人误判。
- 驾驶员获取错误的信息。
- 失去报警。

有效场景机制：

- 夜间、隧道、无路灯或弱照明道路，近光灯无法点亮，驾驶员前方视野不足，未及时发现行人、障碍物或前车。
- 夜间/隧道中近光灯非预期熄灭，驾驶员短时间内视野突然下降。
- 对向来车或前车因异常灯光状态误判本车位置或意图。
- 夜间开灯提示丢失只在驾驶员实际可能忘记开灯且外部照明不足时才有安全相关性。

无效或弱场景：

- 白天正常天气仅近光灯不亮，通常安全影响较弱。
- 把“灯不亮导致违反法规”当作 HARA 人身伤害事件。
- 风险对象不在本车可照明/行驶路径内。
- 把近光灯非预期点亮默认等同于远光灯强眩目；若没有亮度过大或配光错误故障，不要自动推导成高速致命眩目。

## 电子驻车制动 EPB/驻车保持

典型功能：

- 静态开关拉起、释放、自动驻车、坡道驻车、休眠唤醒后拉起。

典型 malfunction：

- 请求拉起时未拉起。
- 拉起夹紧力过大。
- 拉起夹紧力不足。
- 拉起过晚。
- 未请求时拉起。
- 行驶中非预期拉起。
- 卡滞在释放状态导致无法拉起。
- 卡滞在拉起状态导致无法释放。
- 请求拉起却执行释放，或请求释放却执行拉起。
- 驻车状态反馈错误，驾驶员误判车辆已驻稳。

典型整车级危害：

- 非预期的纵向移动。
- 非预期的减速。
- 无法纵向移动。
- 驾驶员获取错误的信息。

有效场景机制：

- 坡道驻车或停车未熄火时，EPB 请求拉起失败，车辆因坡度、驱动蠕行或外力发生非预期移动，撞击行人、后车、障碍物。
- 静止/低速时 EPB 非预期拉起，引起车辆突然减速或后车追尾。
- 行驶中 EPB 非预期拉起属于低可控性高风险场景；若当前功能只允许静态拉起，必须说明非预期激活如何越过操作域。
- EPB 夹紧状态反馈错误可能导致驾驶员离车，车辆随后移动，驾驶员不在车上时 C 往往更差。

无效或弱场景：

- “雨雪导致车辆溜坡”但没有坡度、驱动扭矩、外力或制动力不足作为纵向力来源。
- 车辆状态为驻车却给出 60 km/h 以上车速。
- 静态开关拉起功能的故障被直接扩展为高速巡航故障，除非故障模式明确为非预期激活且解释越域。
- 把 Released 状态请求拉起失败、休眠唤醒失败等不同前置条件重复写成多个功能丧失故障；sheet1 应概括为“驻车时 EPB 无法拉起”，前置条件在 sheet2/HARA 展开。
- 驾驶员主动拉起开关通常没有“过早”故障；请求前拉起属于非预期激活。

## 制动系统

典型危害包括失去减速、制动能力降低、非预期减速。核心场景关注前车距离、速度、路面附着、弯道/直道、后车距离和驾驶员可用反应时间。

## 转向系统

典型危害包括失去横向运动、非预期横摆、方向错误。核心场景关注弯道、变道、车道保持、对向车辆、道路边界和驾驶员施力能力。

## 动力/驱动系统

典型危害包括非预期加速、失去加速、相反方向纵向移动。核心场景关注起步、超车、路口、拥堵跟车、坡道、倒车和驾驶员可通过制动/换挡控制的能力。

## HMI/提示/报警

HMI 本身通常不是直接伤害源。只有当错误信息、误报警、漏报警会改变驾驶员决策并导致车辆进入危险状态时，才形成危险事件。必须说明驾驶员基于错误信息采取了什么动作。

11: 1.2.1 系统工作原理
12: 
13: 在任意电源档位下，通过切换组合开关上的灯光挡位来亮灭近光灯。前车身控制器作为信号采集和处理的主控，接收组合开关发送的灯光档位信号来控制近光灯的亮灭。若为美版车，则仪表显示系统还需要同步点亮灯光总开关指示灯。
14: 
15: 1.2.2 功能逻辑
16: 
17: 1.2.2.1 近光灯亮灭逻辑
18: 
19: ● 输出控制描述：
20: 
21: （1）控制近光灯熄灭：近光灯状态为熄灭，断开近光灯驱动，发送“近光灯”报文（0xXXX，X.X-X.X,0x2）。
22: 
23: （2）控制近光灯点亮：近光灯状态为点亮，输出近光灯驱动，发送“近光灯”报文（0xXXX，X.X-X.X,0x1）。
24: 
25: ● 逻辑描述：
26: 
27: （1）模块得常电需要实时记忆近光灯状态（点亮、熄灭），断电不记忆。
28: 
29: （2）满足下面任意条件，控制近光灯点亮：
30: 
31: （a）近光灯档：接收到1帧“灯光旋钮当前机械结构处于的档位”报文（0x133，4.3-4.5，0x4），或掉线时判断为近光灯档。
32: 
33: （b）AUTO档：根据自动灯光逻辑，判断为近光灯点亮时。
34: 
35: （3）满足下面任意条件，控制近光灯熄灭：
36: 
37: （a）非近光灯/AUTO档：接收到1帧“灯光旋钮当前机械结构处于的档位”报文（0x133，4.3-4.5，0x1/0x3）。
38: 
39: （b）AUTO档：根据自动灯光逻辑，判断为近光灯熄灭时。
40: 
41: （4）掉线处理：连续5s未接收到“灯光旋钮当前机械结构处于的档位”报文（0x133，4.3-4.5）时，认为掉线，执行如下：
42: 
43: （a）ON档电下掉线，若此时采集到近光灯开启/关闭信号为低电平，认为此时处于近光灯档。
44: 
45: （b）ON档电下掉线，若此时采集到近光灯开启/关闭信号为非低电平，默认保持上一状态。
46: 
47: （c）OFF档电下掉线，默认保持上一状态。
48: 
49: （5）整车处于防盗设定状态/防盗状态时，不处理灯光开关的变化（判断“灯光旋钮当前机械结构处于的档位”报文）。
50: 
51: （6）模块上常电默认灯光旋钮的左灯光档当前处于OFF档，近光灯状态为熄灭，断开近光灯驱动，发送“近光灯”报文（0xXXX，X.X-X.X,0x2）。
52: 
53: （7）模块在不掉常电下出现自复位时，近光灯状态需要保持自复位前的状态，然后再按正常逻辑执行。
54: 
55: 1.2.2.2 灯光总开关指示灯逻辑
56: 
57: 若为美版出口车，在控制近光灯亮灭时，需控制仪表显示系统的灯光总开关指示灯同步亮灭。非美版出口车，无此指示灯。
58: 
59: 仪表自身会要求对美版出口车做区分。
60: 
61: 1.2.2.3 夜间行车语音提示功能
62: 
63: （1）同时满足下面所有条件，按照事件周期格式先以短周期发送三帧“夜间行车开灯语音提示”报文（0xXXX，X.X，0x1），然后再发送“夜间行车开灯语音提示”报文（0xXXX，X.X，0x0）：
64: 
65: （a）电源档位为ON档。
66: 
67: （b）灯光开关处于OFF档或小灯档：接收到1帧组合开关发送的“灯光旋钮当前机械结构处于的档位”报文（0x133，4.3-4.5，0x1/0x3）。
68: 
69: （c）车速≥5km/h：接收到1帧“车速信号”报文（0x121，1.0-2.3，车速≥5）。
70: 
71: （d）外界光强档位为5档：发送“光照强度值”CAN报文（0xXXX，X.X-X.X，0x5）时。
72: 
73: （2）不满足上面任一条件（（a）-（d）），发送“夜间行车开灯语音提示”报文（0xXXX，X.X，0x0）。
74: 
75: （3）掉线处理：连续5s未接收到“灯光旋钮当前机械结构处于的档位”报文时，认为组合开关掉线，报文默认保持上一状态；连续10帧未接收到“车速信号”报文时，认为ESP掉线，报文默认保持上一状态。
76: 
77: （4）模块上常电默认发送“夜间行车开灯语音提示”报文（0xXXX，X.X，0x0）。

84: 
85: （a）诊断当前故障有“近光灯驱动电路对地短路故障”。
86:

88:

90: 
91: （a）诊断当前故障无“近光灯驱动电路对地短路故障”。

96: 
97: （4）模块得常电或者休眠唤醒时，默认近光灯正常。
