# Sheet1 Malfunction 思维模型

本指南只用于 `derive_mf` 阶段。sheet1 的任务不是筛选安全风险，而是把当前功能所有逻辑存在的功能故障写出来；安全相关性、整车危害和场景严重性放到后续 sheet 判断。

## 核心判断

只要某个功能故障在当前功能的正常行为模型中逻辑存在，就应写入 sheet1。不要因为暂时想不到危害、ASIL 可能较低、源文档没有明示该故障，就把它从 sheet1 删除。

sheet1 只回答：

```text
这个功能本来应该怎样工作？
它可能以哪些方式偏离这个正常行为？
这些偏离分别属于哪个引导词？
```

sheet1 不回答：

```text
会不会造成事故？
事故发生在哪个道路场景？
ASIL 是多少？
安全目标是什么？
根因是哪个电路/报文/软件？
```

## 第一步：建立功能行为模型

从源文档提取当前功能的行为元素。不要只看功能名称，要看输入、条件、输出和状态。

- 触发源：驾驶员请求、自动条件、远程命令、系统状态变化。
- 允许条件：电源档位、车速范围、系统状态、互锁条件、网络/休眠状态。
- 执行动作：点亮、熄灭、拉起、释放、制动、加速、转向、锁止、提示。
- 目标状态：打开/关闭、Applied/Released、锁止/解锁、启用/停用。
- 物理作用量：亮度、夹紧力、制动力、扭矩、角度、速度、压力、温度、持续时间。
- 时间关系：响应时间、保持时间、自动触发时机、阶段顺序。
- 方向/极性：打开对关闭、拉起对释放、左对右、加速对减速、前进对后退。
- 反馈/提示：状态显示、报警、声音、指示灯；只有它们影响用户决策时才继续进入危害分析。

建立模型后先确定一个 `故障对象`，用于统一 sheet1 描述。故障对象应是用户或车辆层面可理解的功能对象/执行对象，例如 `近光灯`、`EPB`、`驻车制动`、`车窗`、`转向助力`，不要用道路、天气、车速、档位、休眠等条件作开头。

## 第二步：逐引导词做“存在性判断”

对 8 个引导词逐个回答：当前功能行为模型里是否存在这个偏离维度？存在就写；不存在才留空。

同一个异常行为在同一引导词内不要重复。不同引导词从不同角度提问，可能都需要保留：例如 `无法熄灭` 是 required control action 未提供，属于功能丧失；`卡滞在点亮状态` 是状态无法改变，属于卡滞。优先选择更具体的引导词只用于避免同一列内或同一语义下的重复，不用于删除本应由其他引导词覆盖的故障。

```text
卡在某状态无法改变 -> 卡滞
请求 A 却执行 B -> 方向错误
没有请求或不满足条件却动作 -> 非预期激活
请求后响应慢 -> 过晚
作用量太大/太小 -> 过大/过小
应提供的控制动作或目标状态没有提供 -> 功能丧失
```

### 功能丧失

判断问题：在应提供某个功能动作、控制动作或目标状态时，是否可能没有提供？

写功能级偏离，不写前置条件组合。

功能丧失不是只看“启动/打开/拉起”方向，而是看当前子功能负责的 required control actions 是否缺失。对于双向状态控制功能，如果当前功能同时负责 A 和 B 两个动作或状态转换，应分别考虑 `无法A` 和 `无法B`。例如近光灯功能同时负责点亮和熄灭，因此 `近光灯无法点亮` 与 `近光灯无法熄灭` 都属于功能丧失候选。

对于单向子功能，只分析当前子功能负责的动作，不因为系统存在反向功能就扩展。例如 `静态开关拉起` 只分析拉起，不自动生成释放功能的功能丧失。

功能丧失与卡滞的边界：

- 功能丧失回答“请求/条件成立时，目标控制动作是否没有提供”。
- 卡滞回答“功能状态或执行器是否保持在某个状态无法改变”。
- 如果当前功能是双向状态控制，两者可以同时存在，但描述角度必须不同：`近光灯无法熄灭` 与 `近光灯卡滞在点亮状态` 可以同时保留；不要在同一引导词中重复写同一本质条件变体。

好：
- `EPB无法拉起`
- `近光灯无法点亮；近光灯无法熄灭`

差：
- `车辆静止且 EPB 处于 Released 状态时无法拉起；整车 OFF 或网络休眠状态下唤醒失败无法拉起`

前者是同一故障在不同条件下的场景化展开，应留到 sheet2/HARA。

### 过大

判断问题：功能输出、作用量、强度、持续时间或效果是否可能超过预期？

存在物理作用量就要思考过大，即使功能表面上是开/关、拉起/释放。开关动作背后常有亮度、力、扭矩、制动力、保持时间等作用量。

如果当前功能的最终车辆级输出本身具有强度或效果大小，例如灯光照明、制动/驻车、转向、驱动、车窗/尾门运动、加热/冷却，则不要因为源文档只写了“驱动开/关”“执行拉起”就忽略作用量。只有功能对象确实只是纯逻辑信号、且不会直接形成车辆物理输出时，才可以不写过大。

示例：
- 照明功能可能有亮度/照度过大。
- 驻车/制动功能可能有夹紧力或制动力过大。
- 转向功能可能有转角或助力过大。

### 过小

判断问题：功能输出、作用量、强度、持续时间或效果是否可能不足？

过小与过大成对审查。只要功能存在车辆级物理输出，通常也应存在输出不足/效果不足这一偏离；不要只写功能丧失而漏掉“还能输出但不足”的中间状态。

示例：
- 照明功能可能有亮度/照度过小。
- 驻车/制动功能可能有夹紧力或制动力不足。
- 动力功能可能有扭矩过小。

### 过早

判断问题：功能是否有独立于用户即时请求的预期触发时刻、自动时序或阶段顺序？如果有，是否可能在正确时机之前发生？

主动离散请求本身通常没有“过早”。如果功能在没有请求前发生，通常归入 `非预期激活`，不是 `过早`。

只有当前功能自身拥有自动触发时机或阶段顺序时，才写过早。如果文档只是说“根据某上游自动逻辑判断后执行”，而该上游逻辑不是当前功能清单中的功能，则上游判断过早不要写入当前功能 sheet1；当前功能只分析收到请求/条件后的执行偏离。

好：
- 自动灯光在环境仍明亮时过早点亮。
- 自动释放在允许条件满足前释放。

通常不写：
- 驾驶员按开关打开/关闭。
- 驾驶员拉起 EPB 开关。

### 过晚

判断问题：请求或触发条件已经成立后，功能是否可能响应延迟、结束延迟或状态建立延迟？

示例：
- 点亮响应过晚。
- 熄灭响应过晚。
- 拉起/释放响应延迟。
- 制动力建立延迟。

如果同一引导词下存在两个不同动作或状态方向，不要用 `/` 压缩，例如不要写 `近光灯点亮/熄灭响应过晚`。应按与其他引导词一致的粒度用分号列出：`近光灯点亮响应过晚；近光灯熄灭响应过晚`。

### 非预期激活

判断问题：没有请求、不满足允许条件、或应保持当前状态时，功能是否可能发生状态改变或动作输出？

对于双向状态功能，要思考两个方向：

- 不该打开时打开。
- 不该关闭时关闭。
- 不该拉起时拉起。
- 不该释放时释放。

这些可以在同一引导词单元格内用同一 MF 编号分号分隔，后续 sheet2 再拆分。

### 卡滞

判断问题：功能状态、输出或执行器是否可能保持在某个状态，无法按请求改变？

必须写明卡滞状态，不能只写“状态卡滞”。

示例：
- 卡滞在打开状态。
- 卡滞在关闭状态。
- 卡滞在拉起状态。
- 卡滞在释放状态。

### 方向错误

判断问题：功能是否存在相反方向、相反状态或相反效果？如果存在，请求 A 时是否可能执行 B？

方向错误与非预期激活的边界：

- 有请求 A，但系统执行了相反动作 B -> `方向错误`。
- 没有请求、不满足允许条件、或应保持当前状态却发生动作 -> `非预期激活`。

对于单向状态转换功能，`请求当前动作却执行相反动作` 属于当前功能的方向错误，因为它描述的是当前请求到输出的映射错误；不要因为相反动作通常由另一个功能负责就删除该方向错误。相反地，`卡滞在目标状态导致无法执行反向功能` 通常属于反向功能，这只影响卡滞归属，不影响方向错误判断。

示例：
- 请求打开却关闭。
- 请求关闭却打开。
- 请求拉起却释放。
- 请求左转却向右转。
- 请求加速却减速。

## 第三步：控制 sheet1 粒度

sheet1 描述“故障是什么”，不描述“在什么场景发生”。使用以下粒度：

- 只分析当前 `子功能` 负责的行为偏离。系统存在相反状态，不代表当前单向子功能要承担相反功能的故障。
- 对单向状态转换功能，起始状态卡滞导致无法达到当前功能目标，属于当前功能的卡滞故障；目标状态卡滞导致无法执行相反功能，通常属于相反功能。例：“静态开关拉起”负责从释放到拉起，应保留 `卡滞在释放状态无法拉起`，但 `卡滞在拉起状态无法释放` 通常属于释放功能，除非源文档把释放也列入当前子功能。
- 同一功能偏离，不因电源档位、状态、休眠唤醒、道路、天气、人群、车速不同而拆成多个 sheet1 故障。
- 同一本质故障不要因操作域或允许条件不同而拆成多个 sheet1 故障。若两个描述只是在“未请求、行驶中、静止/低速、车速超过门限、休眠、OFF/ON、Released/Applied、坡道、夜间”等条件上不同，而偏离动作相同，应保留一个功能级故障；这些条件放到 sheet2/HARA 的场景中展开。
- `非预期激活` 优先描述状态改变本身，例如 `EPB非预期拉起`、`近光灯非预期点亮`。不要把 `未请求拉起`、`行驶中拉起`、`不满足条件时拉起` 写成多个故障，除非它们代表不同的动作方向或不同的功能行为偏离。
- 不同偏离方向要拆开或在同一引导词下列出，例如非预期打开和非预期关闭。
- 不同物理作用量要拆开，例如夹紧力过大和夹紧力不足。
- 根因、检测机制、报文、诊断、电路、软件策略不要放入 sheet1。
- 如果某故障存在但后续没有安全影响，仍可在 sheet1 保留；sheet2 可映射为 `无危害` 或在备注说明。

## 第四步：统一故障描述风格

sheet1 的每个故障描述都应使用同一种短语口径：`功能对象 + 偏离表现`。同一行不要混用“场景条件句”“请求句”“短语句”等多种风格。

推荐模板：

- 功能丧失：`<功能对象>无法<关键动作>`。
- 过大：`<功能对象><作用量>过大`。
- 过小：`<功能对象><作用量>过小/不足`。
- 过晚：`<功能对象><动作>响应过晚`。
- 非预期激活：`<功能对象>非预期<动作/状态改变>`。
- 卡滞：`<功能对象>卡滞在<状态>`。
- 方向错误：`<功能对象>请求<动作A>却<动作B>`。

如果确实需要条件才能避免歧义，只保留最小功能条件，不写道路、天气、人群、车速、电源档位等场景化条件。功能对象可以使用系统通用对象名或当前功能对象名，但同一功能行内应保持一致。

用词尽量与引导词一致：`过晚` 写 `响应过晚`，不要混写 `延迟`；`功能丧失` 写 `无法...`，不要写成危害后果；`方向错误` 必须同时出现请求动作和实际相反动作。

## 输出前自检

1. 是否先建立了触发源、条件、动作、状态、作用量、时序、方向的功能行为模型？
2. 每个非空故障是否都是功能行为偏离？
3. 是否把“是否安全相关”提前用于删减 sheet1？如果是，恢复该故障。
4. 是否漏掉作用量的过大/过小？
5. 是否把无请求发生误写成过早？
6. 是否把同一偏离按前置条件重复写成多个故障？
7. 是否把非预期激活按“未请求/行驶中/不满足条件”等操作域重复拆成多个故障？
8. 双向状态控制功能的功能丧失是否覆盖了两个 required control actions，例如点亮和熄灭、打开和关闭？
9. 同一功能行的故障描述是否都采用“功能对象 + 偏离表现”的统一风格？
10. 同一引导词内是否用分号拆分不同动作/状态，而不是用 `/` 压缩？
11. 如果存在相反状态/动作，是否检查了 `请求 A 却执行 B` 的方向错误？
12. 卡滞是否写明状态？
13. 方向错误是否写明请求和实际相反？
