# SEC 场景扎根审计

目标：审计 S/E/C 评级和解释是否由当前单条危险事件自然推出。审计者不能知道其他场景、用户期望答案或后处理规则。

## 审计原则

1. 场景事实优先：只使用本行 `道路类型`、`道路条件`、`环境条件`、`车辆状态`、`车速(km/h)`、`特殊要素`、`附加条件`、`危害事件`、`有风险的人员`、故障描述和整车危害。
2. 禁止补事实：SEC 解释不得引入场景中没有的时序、操作状态、环境、风险对象或驾驶员行为。若解释中出现“起步、傍晚、进入隧道、主动开灯、未及时发现、距离较近、行人横穿”等事实，必须能在本行场景字段或附加条件中找到对应依据。
3. 评级从场景推出：不要为了让 ASIL 变高或变低而改写 E/C/S 的理由。若现有评级不合理，应直接给出更合理的评级和理由。
4. 不按功能套模板：即使某个功能族有常见经验，也必须转化为当前场景的因果判断。不同系统、不同故障、不同场景都按同一审计方法处理。
5. 低风险也要可解释：QM 不是默认结论。若评级为 QM，必须说明是 S、E、C 中哪一项使风险降低，并且该降低原因来自场景事实。
6. 功能族知识只用于物理机制审查：可用于判断某故障是否真的能造成该危害、风险对象是否在影响路径内、可感知性是否合理；不能直接当成固定 E/C/S 结论。
7. 禁止假设性旁支：SEC 解释不得写“若存在/若无/可能还有”等本行未给出的替代条件。若需要某条件才能成立，应在本行场景中已有该条件，否则应删除该条件或把评级按现有场景重做。
8. 反模式优先：若当前 SEC 或危险事件违反 prompt 中的 HARA 评审反模式，`approved` 必须为 `false`，并在 `final_sec` 中按本行事实给出更合理的评级。不得因为原场景写得严重就顺着错误场景给高评级。

## S/E/C 审计步骤

### 1. 审计 E

判断 `E-解释` 是否描述当前操作场景的暴露频率，而不是故障概率或模板化场景。

合格的 E 解释应包含：

- 当前场景的核心暴露条件，例如道路类型、环境、车辆状态、速度范围、特殊要素或附加条件中的限定条件。
- 为什么这些条件组合是 E0/E1/E2/E3/E4。
- 若使用“低概率子场景”，必须指出子场景来自附加条件，而不是临时补充。

不合格示例类型：

- 场景为“夜间前向匀速”，E 解释却说“起步/傍晚开灯时”。
- 场景没有“行人横穿”，E 解释却以“学校区域行人横穿”作为暴露依据。
- E 解释把“故障发生概率低”当作暴露频率低。

### 2. 审计 S

判断 `可能的后果('S'的理由)` 是否说明风险对象、碰撞或伤害机制、速度/能量依据。

合格的 S 解释应包含：

- 谁受伤。
- 如何受伤，例如追尾、侧碰、撞击障碍物、撞击行人、车辆溜车撞击。
- 速度、相对速度、坡度/距离/能量等依据。

### 3. 审计 C

判断 `C-解释` 是否与场景中的可感知性、反应时间、可用操作和空间约束一致。

合格的 C 解释应包含：

- 异常如何被驾驶员或相关人员感知。
- 反应时间或发现时机是否充分。
- 可用操作，例如制动、转向、停车、切换模式、重新操作、避让。
- 场景是否限制这些操作，例如车速高、距离近、弯道、坡道、驾驶员离车、风险对象突然出现。

不合格示例类型：

- 场景写明“驾驶员未及时发现”，C 解释却说“驾驶员会立即发现”。
- 驾驶员离车场景按驾驶员可踩刹车处理。
- 高速近距离或弯道场景按充足反应时间处理，但没有说明可视距离、制动距离或道路空间。
- 场景没有对向来车，C 解释却写“若无对向来车可开启远光灯”。
- 场景是多车道隔离/封闭道路，解释却引入对向会车或正面碰撞机制。

### 4. 审计功能物理机制

使用当前 prompt 中的功能族物理知识检查：

- 故障是否是危险事件的必要原因。
- 危害路径是否符合该功能的物理作用范围。
- 风险对象是否真的位于该功能影响路径内。
- 可感知性和可控性是否符合该功能的典型表现。

如果功能族知识与当前 SEC 冲突，不要机械套用功能族结论；先回到本行场景，说明当前场景为什么适用或不适用该知识。

### 5. 审计危险事件是否被过度放大

SEC 复核不仅检查解释文字，还要检查当前危险事件是否把弱故障机制放大成不合理后果。若危险事件本身已经包含夸大的碰撞路径，但本行事实和功能物理知识不支持该路径，应在 S/C/E 中按合理机制评级，并在解释中使用合理机制，不继续放大。

常见审计问题：

- 弱感知干扰被写成高速正面碰撞，但没有亮度、配光、失控或车道侵入等支撑。
- “短暂延迟”被当成长期完全失效，但场景没有说明延迟窗口内已经进入不可避危险。
- 道路拓扑不支持风险对象，例如封闭快速路默认行人横穿。
- C 评级把“可能制动/减速/停车”完全忽略，或在距离、速度、空间明显受限时又写成完全容易控制。

## 修订要求

如果现有 SEC 通过审计，保持原评级和解释，只做必要的中文精炼。

如果现有 SEC 未通过审计，重新给出整组 `E-解释`、`暴露频率'E'`、`可能的后果('S'的理由)`、`Severity 'S'`、`C-解释`、`控制能力 'C'`。修订必须：

- 只使用本行事实。
- 不出现“评级校正”“修正”“按规则压低/提高”等后处理措辞。
- 不把审计发现写成输出解释。
- 不为了保留原 ASIL 而牺牲场景一致性。
