# HARA 场景条件一致性审查框架

HARA 场景不是若干高风险词的拼接。每一行 HARA 都必须先通过“场景条件是否能同时成立”的审查，再讨论 S/E/C。

## 通用审查维度

### 1. 运动学一致性

- `车辆状态`、`车速(km/h)`、`驾驶员是否在车上` 必须能同时成立。
- 驻车、停车熄火、停车未熄火通常只能对应静止或极低速蠕行。
- 前向匀速、变道、超车、自动巡航、智能领航不能对应静止车速。
- 泊车、自动泊车、前向爬行通常不应使用中高速车速。
- 如果驾驶员不在车上，C 解释不能依赖“驾驶员踩刹车、重新操作、立即制动”等动作。

### 2. 道路拓扑一致性

- 高速公路、城市快速路等封闭或快速通行道路，不应默认存在行人横穿、骑行者、红绿灯路口、环岛路口。
- 若高速/快速路场景确实涉及弱势交通参与者，必须写明施工、事故、服务区、收费站、应急车道、路肩、救援、养护等特殊前提。
- 单行道、多车道隔离道路不应默认出现对向来车、迎面碰撞、会车或正面碰撞。
- 隧道、匝道、坡道、路口等特殊道路要素如果出现在危害事件或理由中，也应体现在 `特殊要素` 或 `道路条件` 中。

### 3. 环境一致性

- 场景文字写夜间、黑暗、无路灯、低照度时，环境条件通常应为夜间；日间隧道可以例外，但必须说明隧道内光照。
- 场景文字写白天、日间、环境光充足时，环境条件不能同时为夜间。
- 降雨、雨后、积水、湿滑、浮水应由 `环境条件=降雨(小/大/暴)` 或 `道路条件=浮水路面` 支撑。
- 降雪、冰雪应由 `环境条件=降雪(小/大/暴)` 或 `道路条件=冰雪路面` 支撑。

### 4. 风险对象路径一致性

- 有风险的人员必须位于危险路径内，并且能出现在该道路场景中。
- `行人数量(无)` 不能同时写行人受伤或行人误判。
- 追尾/撞击前车需要前方车辆、慢行车辆、停驶车辆、与前车距离、拥堵等支撑条件。
- 对向车辆风险需要道路允许对向交通；单行道或多车道隔离场景不能默认存在对向车辆。

### 5. 故障-危害-事件机理一致性

- 危险事件必须体现当前故障如何导致当前整车危害，不能偷换成另一种危害。
- `驾驶员视野丢失或者降低` 应体现照明、可见距离、识别、发现、低照度等机理。
- `其他驾驶员/行人误判` 应体现眩目、位置/距离/意图判断干扰，不应改写为本车驾驶员视野不足。
- `非预期的纵向移动` 必须有坡度、蠕行、外力、驱动扭矩或已有保持力释放作为纵向力来源。
- `非预期的减速` 应发生在行驶或低速移动状态；静止驻车下通常是无危害或舒适性/零件影响。
- `驾驶员获取错误的信息` 必须说明错误信息如何改变驾驶员行为并导致危险。

### 6. 坡道方向与碰撞对象一致性

在没有特殊说明时，`上坡道路/下坡道路` 默认按车辆前进方向理解：

- `下坡道路` 上失去驻车保持，车辆应沿坡度方向向前滑移或向前溜车，风险对象通常在车辆前方：前车、前方行人、前方障碍物、路口或主路车辆。
- `上坡道路` 上失去驻车保持，车辆通常沿坡度方向向后滑移或向后溜车，风险对象通常在车辆后方：后车、后方行人、后方排队车辆或后方障碍物。
- 如果与上述方向相反，必须明确说明原因，例如车头朝上坡/下坡的特殊朝向、倒车方向、外力反向推动、D挡驱动蠕行或驱动扭矩克服坡度。
- 危害事件、附加条件、风险对象、S理由和C理由中的前/后方向必须一致。不能写 `下坡道路` 却说车辆向后滑移撞击后方行人；也不能写 `上坡道路` 却说车辆向前滑移撞击前车，除非有驱动扭矩或外力支撑。
- `追尾` 表述也要符合相对方位：本车向前撞前车可称追尾，后车撞本车可称被追尾；本车向后溜移撞后车/后方车辆不应写成“追尾后方车辆”，应写“撞击后方车辆”。

## 生成前自问

对每个 HARA 场景，先回答：

1. 这些道路、环境、车速、车辆状态能同时出现吗？
2. 风险对象是否真的能位于车辆危险路径内？
3. 没有当前故障时，这个危险事件是否会消失或显著降低？
4. 危害事件描述是否和 sheet2 的整车危害是同一种机理？
5. 如果是坡道纵向移动，滑移方向和前/后风险对象是否与坡道方向一致？
6. S/E/C 理由是否只使用本行已经给出的场景事实？
