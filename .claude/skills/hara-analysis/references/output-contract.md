# 输出契约

最终写入一个 JSON 文件，然后由 `scripts/hara_tool.py export` 生成 Excel。

## JSON 顶层结构

```json
{
  "project": {
    "system_name": "EPB",
    "source_file": "EPB.docx",
    "item_definition": "系统边界和功能摘要",
    "source_functions": ["静态开关拉起"],
    "assumptions": ["缺失信息或保守假设"]
  },
  "derive_mf": [],
  "mf_vehicle_hazards": [],
  "HARA": [],
  "sg_sum": []
}
```

sheet 数组内对象的键名必须与下列字段完全一致。

## sheet1: derive_mf

字段顺序：

- `No.`
- `子功能`
- `功能丧失`
- `过大`
- `过早`
- `过小`
- `过晚`
- `非预期激活`
- `卡滞`
- `方向错误`

规则：

- `No.` 为 `系统名_fc编号`，例如 `EPB_fc01`。
- `子功能` 必须优先来自源文档功能清单表中的功能名称。若源文档功能清单只有一个功能，例如 `近光灯`，则 sheet1 通常只有一行，不要把“亮灭逻辑、语音提示、故障检测”拆成多行，除非功能清单明确列出它们是独立功能。
- 每个非空引导词值以 `MF` 编号开头，例如 `MF101 近光灯无法按驾驶员请求点亮`。
- 编号规则：`MF + 功能序号 + 当前功能中非空引导词序号`。例如第 1 个功能的第 1 个非空引导词是 `MF101`，第 1 个功能的第 2 个非空引导词是 `MF102`。
- 一个单元格可包含同一 MF 编号的多个故障，用 `；`、`;`、`或` 分隔，但 sheet2 必须拆分成独立条目。
- sheet1 故障描述只写功能级偏离，不写根因、场景、危害后果或安全机制。
- sheet1 不按安全相关性过滤故障。只要故障在当前功能行为模型中逻辑存在，就应写入；后续可映射为 `无危害` 或 QM。
- 故障描述粒度保持在功能级：同一偏离不要按电源档、状态、休眠唤醒、道路、天气或车速重复拆分。
- 每个引导词是否适用，应由当前功能的触发源、允许条件、执行动作、目标状态、作用量、时序和方向关系推导。

## sheet2: mf_vehicle_hazards

字段顺序：

- `No.`
- `Milf_ID`
- `故障描述`
- `整车级危害`
- `备注`

规则：

- `Milf_ID` 为 `系统名_Milf_编号`，例如 `EPB_Milf_001`。
- `故障描述` 必须复用 sheet1 中的 MF 编号和具体故障描述。
- `整车级危害` 只能来自 `assets/vehicle_hazards.json`。

## sheet3: HARA

字段顺序：

- `List_No`
- `MF_ID`
- `故障描述`
- `整车危害`
- `道路类型`
- `道路条件`
- `环境条件`
- `车辆状态`
- `车速(km/h)`
- `特殊要素`
- `附加条件`
- `驾驶员是否在车上`
- `危害事件`
- `E-解释`
- `暴露频率'E'`
- `有风险的人员`
- `可能的后果('S'的理由)`
- `Severity 'S'`
- `C-解释`
- `控制能力 'C'`
- `结果ASIL`
- `安全目标`
- `安全状态`
- `FTTI(ms)`

规则：

- `MF_ID` 为 `系统名_MF_编号`，例如 `EPB_MF_01`。
- `故障描述`、`整车危害` 必须和 sheet2 对应。
- 场景字段必须来自 `assets/operation_scenarios.json`。
- 每个非 `无危害` 的 sheet2 故障通常应有不少于 3 条 HARA 场景，覆盖最高严重度、高暴露、低可控性/边界工况等不同风险机制；复杂故障应有 4-6 条。不能只给一两条场景就进入 SEC。
- `暴露频率'E'` 取 `E0/E1/E2/E3/E4`。
- `Severity 'S'` 取 `S0/S1/S2/S3`。
- `控制能力 'C'` 取 `C0/C1/C2/C3`。
- `结果ASIL` 必须由脚本根据 S/E/C 计算。
- ASIL 为 `QM` 时，`安全目标`、`安全状态`、`FTTI(ms)` 为空。

## sheet4: sg_sum

字段顺序：

- `SG_No`
- `安全目标`
- `ASIL Level`
- `安全状态`
- `操作模式`
- `FTTI(ms)`

规则：

- `SG_No` 为 `SG001` 形式。
- 每个故障 MF 在 HARA sheet 中取最高 ASIL 条目；最高 ASIL 不为 `QM` 时，该 MF 必须形成一条 `sg_sum`。
- 不同故障 MF 不得因为 `安全目标` 文字相同或相似而合并；最高 ASIL 为 `QM` 的故障 MF 不生成 `sg_sum`。
- `ASIL Level` 取该 MF 的最高 ASIL。
- `安全目标`、`安全状态`、`FTTI(ms)` 应来自该 MF 的最高 ASIL 代表条目；同一 MF 的非 QM HARA 行应继承该代表条目的 SG 字段。
- `操作模式` 取 `驻车`、`行车`、`驻车和行车`。
